## 第二轮 · CTO：一场没有八股的技术交流

这一轮没有线程进程、没有八股，全程是"你到底懂不懂你在做的东西"的深度对话。因为我了解这个赛道，反问环节我们直接聊到了同一张桌子上。

### 2.1 反问：我先把赛道质疑甩给了 CTO

我一向主张[反问要有攻击性](/blog/20260528---agentmockinterviewreview/post)，这场我一上来就把最尖锐的赛道问题抛了出去。

**我问：** 你们这个产品还在内测，是从传统工具业务转型做 AI 的一次试水，方向是"AI 版 Canva"。但文档协作这类传统产品市场天花板比较低，而设计这条赛道——Canva、Figma 本身就有巨大的用户量，还在不断揭露 AI 功能；更 AI-native 的设计工具出来之后，Figma 无论股价还是舆论都被打得挺惨。**所以——为什么还要进这个赛道？还是你们有不同的看法？**

**答（CTO）：** Figma 和那类纯 AI 工具之间，会有一个**中间市场**。

- **左侧**像 Canva：海量用户数据、很大的工程团队（可能 2000 多人），追求更低价格、可编辑的设计；
- **右侧**像 Playground、Lovart：价格更高（AI 成本高），但产物**不可编辑**；
- **Figma** 从第一天就面向专业设计师。

我们的目标用户是**非专业设计师**——大家都有设计需求，但能力满足不了自己。HR 想做招聘海报，得去找内部设计师；运营、个体户、老师、学生，多多少少都有这种需求。我们就是满足这群人的，所以和 Figma 很不一样。

**我追问：** HR 要去找专业设计师，这到底是工具问题，还是审美（taste）问题？我自己觉得现在很多人 vibe coding 抽出来的前端质量差，不是模型的问题，更多是个人 taste 的问题——市面上很多在卖、在营销 taste skill 的；还有些产品就固定几套模板，我觉得那种活不长久。

**答（CTO）：** 我们还没到 taste 那一步，离它还有点远。HR 找设计资源，更多不是想更好看，而是一个 **0 到 1** 的过程——没有合适的工具，怎么做出一个符合固定要求的海报。更重要的是**品牌一致性**：公司有自己的 logo、设计系统、设计语言，对外宣传的字体长什么样。满足了设计语言之后好不好看，那是第二步——现在第一步都还没有。

> 💡 **复盘：这段反问为什么"对了"。** 我没问"贵公司主营业务是什么"这种暴露无知的问题，而是带着对赛道的判断去挑战——直接把"Figma 都被打惨了你们凭什么进场"摆上台面。一个对自家定位说不清的团队可以直接 pass；而对方能立刻给出"中间市场 / 非专业设计师 / 品牌一致性 / 0→1"这套差异化叙事，本身就是值得看的信号。**反问的质量，取决于你面试前对这个赛道做了多少功课。**（剧透：这个质疑我当时仍不完全满意，所以三面又问了联创一次，见后文。）

### 2.2 求职规划：我为什么不读研、不去大厂

**问：** 你还没毕业，对自己有什么求职规划？看重什么？

**答（我）：** 本科毕业就想直接就业，不打算读研。无论在前一段实习做 Agent、还是做自己的个人项目、写技术博客，我都觉得"做产品"是件有意思也有意义的事。

毕业后我想做的，姑且叫 **Agent 开发**。我对这个词的理解可能和大家不太一样：从 prompt engineering，到 context engineering，再到 agent、harness……名词一直在变，但**产品形态从以前的 SaaS 变成了现在的 agent，只要我一直在做这件事就好**。

更具体一点，我希望成为一个**产品工程师**。只会写代码的开发在这个时代可能会被淘汰。这也是我**抵制去大厂**的原因——大厂里开发更像螺丝钉，KPI 跟 PM 挂钩，需求有没有人用都由 PM 兜着；而在小厂或初创，我得身兼数职，把"会开发 + 懂用户 + 会增长"这条完整链路跑通。这才是我觉得有挑战、有意思的地方。

> 💡 **复盘：这段我讲得有点散。** 如果重来，我会先用一句话收口：**"我想做产品工程师——会开发，但更关心需求是否真实、产品如何被用户用起来；所以我选初创而不是大厂。"** 再用项目去支撑。**先给结论，再给证据**，比顺着想到哪说到哪要稳。

### 2.3 产品协作：我和创始人吵过的那一架

**问：** 之前实习里，你在产品上做的事有什么印象深的？

**答（我）：** 在前两段实习我们都不太分前后端，每个人都要想点产品的活，leader 给的任务常常是**高度抽象**的——比如做海外产品，我得先 book 竞品 demo、做产品调研，才落到开发。

举个具体的：在一段早期初创实习（fAIshion，一个云试穿平台），当时 CEO 要去给 VC 做 pitch，希望核心功能马上上线。但我的判断是——那时 landing page 和 SEO 都很烂，不如先把这些基础做好、有了稳定流量再开发核心功能；而且核心功能开发周期长，还需要先发问卷确认它是不是用户真正要的。那时候就有了争执。

**问：** 做产品会比做技术更容易吵？

**答（我）：** 也不能这么说。准确说是：拿到完整 PRD 按部就班的开发不会吵；但当开发也分担了一部分 PM 的活，**自由度高了自然会激发更多讨论**，不一定是争执。这很考验 leader——扁平化放权的最大难点是容易发散，需要一个主心骨把方向带正。

### 2.4 技术拷打：从零做一个下拉菜单组件

这是整场最硬核的一段，也是我**当场翻车又靠 UX 知识扳回来**的一段。

**问：** 假设让你来做这个产品最初的 UI 系统 / 组件库。比如点一个"……"按钮弹出下拉菜单——你会怎么做？有什么坑？

**答（我）：** 前端我基本都用 React，生态比 Vue / TanStack 好太多。这种下拉一般有成熟、可商用的开源组件库，build MVP 直接用，视觉一致性还高。现在也不像早期 shadcn 只有五六个主题色，品牌是紫色的，调个紫色主题很容易。

下拉菜单这里关键看放多少内容。**如果一个下拉超过七八个、十来个选项，不如做成可横向移出的侧边栏**——因为下拉是有层级的，会盖住下方同层内容；从用户习惯讲选项应该少，点开、点一个就走。这里还要考虑二级、三级菜单。

接下来是我专门学 UI/UX 时记住的一个细节——**安全三角（safe triangle）**：

> 如果你用过 GCP 的下拉菜单，会觉得很难用、很别扭，但说不出为什么。其实就是因为它是多级菜单，却没做"安全三角"判定：操作子菜单时鼠标稍微一晃出了路径，二级菜单就关了，得重新点。
>
> 安全三角是：点开一级后，**鼠标当前位置会和二级菜单的上下边缘构成一个三角形区域**；只要鼠标还在这个三角形内移动（哪怕斜着往子菜单挪），就**不关闭**子菜单；只有移出三角形才隐藏。

**问：** 一直停在这个三角形里会关闭吗？

**答（我）：** 不会。安全三角就是设一个**缓冲阈值**，让用户在一二级菜单间移动时不会误关子菜单。

**问：** 那如果用户**特别慢**地在三角形范围内移动到菜单，会关吗？

**答（我）：** 这不取决于鼠标快慢，而取决于你**有没有做动态的三角形阈值检测**——只要在三角形内就不关。这个我之前复刻过、有动画演示，可以屏幕共享给你看。（最早 Amazon 那个书店导航就有这个。）

到这里都还好。接下来翻车了——

**问：** 这个 dropdown 要和 trigger 按钮贴在一起，这个定位你会怎么做？

**答（我）：** 你点按钮，它就在按钮底下出来——这不都是现成组件解决的吗？

**问：** 如果让你**从零实现**呢？

**答（我）：** 🔧 那我确实不知道，愿意搜一下（用 Codex 或 Claude）。大概知道是绝对定位写死在按钮下方，外层父容器写 `relative`，再用 `::after`、加一点 `margin-top` 间隔，水平方向 left 对齐……

> 🔧 **这一段我答错了，下面是正确答案。**
>
> 我把"定位浮层"和"画一个小箭头"混在一起胡诌了（`::after`、`margin-top` 那套根本不是定位浮层的方式）。从零实现一个能贴着 trigger、又不被裁切的下拉浮层，正确的思路链是：
>
> 1. **不要只靠 `position: absolute` 写死。** 父 `relative` + 子 `absolute` 只能解决最简单情况，但按钮靠近视口边缘时菜单会**溢出屏幕**；父级只要有 `overflow: hidden` 或 `transform`，菜单就会被裁切或定位错乱。
> 2. **用 Portal 把浮层渲染到 `document.body`。** 这样浮层脱离父级的 `overflow` / `z-index` 堆叠上下文，不会被裁切。代价是位置得**手动计算**。
> 3. **用定位库做计算——[Floating UI](https://floating-ui.com/)（Popper.js 的继任者）。** 给它 reference（按钮）和 floating（菜单），它算坐标，并提供 middleware：`offset`（间距，我现场想说的"margin-top"其实就是它）、`flip`（下方放不下自动翻上方，**碰撞检测**）、`shift`（左右快溢出时推回视口）、`size`（限高、超出滚动）。
> 4. **这正是 shadcn / Radix 底层在做的事**——[Radix 的 Popper](https://www.radix-ui.com/primitives/docs/components/dropdown-menu) 内部就是 Floating UI + Portal + 碰撞检测。所以更好的答法是："成熟组件库之所以好用，正是因为它帮你封装了 Portal + Floating UI 的定位、flip/shift、focus trap、键盘导航和 ARIA——这些从零写非常容易出 bug。"
> 5. **顺带提一句 2026 的新解法：原生 CSS。** 现在浏览器有 [CSS Anchor Positioning](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_anchor_positioning)（`anchor-name` / `position-anchor` / `position-area`）+ [Popover API](https://developer.mozilla.org/en-US/docs/Web/API/Popover_API)，能把"锚定到 trigger""渲染到顶层不被裁切""自动 flip"这些**纯 CSS/HTML 就做掉一大半**。面试里能主动提这个是明显加分。
>
> **复盘心法：被问"从零实现"时，承认不知道 + 愿意搜没问题；但更好的是把它拆成"我知道边界条件是什么"——溢出裁切、碰撞翻转、堆叠上下文——哪怕答不出具体 API，能说清"难点在哪"也是理解的证明。** 我当场只做到了承认不知道，却又硬编了一套错的实现，这是最该避免的。

### 2.5 Agent 怎么做出"有品味"的设计

**问：** 我在做一个类似 Canva 的 agent，你觉得 agent 要怎么设计出**有品味**的设计？

**答（我）：** 市面上常见做法是用 **skills**，有很有名的所谓 **taste skill**，类似预制提示词，把有 taste 的人的经验封装进去。如果不靠 skill：

- **模型选择很重要。** 有些模型对前端有强偏好依赖，经典的就是 Codex 很爱做卡片式设计。我说的"依赖"是指你想 **zero-shot 抽卡**；如果已有现成项目、或指令非常精确，各家模型都做得不错。难的是"抽卡"。
- **能沉淀成 design system 最好。** 如果目的是贴合公司上下文（品牌调性、字体资产），把这些归纳成 design system 很有价值。我自己就在做：把主项目的 `design.md` 应用到子项目上，让用户一看就知道是同一家的东西，从而互相引流。
- **字体这种精确信息，直接硬编码进 markdown。** 我发现截一段字体的图去问 AI 是什么字体，识别率很低。所以字体名、字号、weight、主题色 / 副色 / 字体色这些直接写死在 markdown 里，省心也省 token。
- **真要有品味，得先让它见过好东西。** 要么 RL / fine-tune（很难），要么喂一批好素材（数量要控制）。
- **让它像 agent 一样自我审核（self-review loop）。** Claude Code、Codex 都内置 preview，可以用无头浏览器截图（Claude Code 用的应该是 Playwright screenshot）。因为光看代码只能看出对不对、有没有乱码，**相对位置和颜色只能靠脑补**——不如让 AI 截图、再用多模态看哪里不好、重新试。我自己做过：让 OpenClaw 装 Playwright screenshot，丢个简单需求，它自己反复三轮——做、截图、多模态自检、再改——差不多了再交还给我。

**问：** 除了 Claude 和 GPT 系列，你直觉哪个模型比较胜任？

**答（我）：** **纯抽卡的话，我觉得 Gemini（Google AI Studio）最强**——不靠额外 skill，直接说"帮我做一个什么产品的 demo / landing page"。微调前端小样式时，谷歌的 **Flash** 系列对 prompt 的理解也最到位——别的模型要写很清楚，Flash 不用。Flash 在这种小修改任务上，性价比、速度、质量我觉得都是最优解。

> 💡 **复盘：可以再补一刀。** 对"做产品的 agent"最关键的点我现场没串起来：**有品味 ≠ 一次抽卡抽得好，而是"可复现地贴合品牌"。** 对 Canva 这类产品，用户要的不是"惊艳但每次都不一样"，而是"每次都稳定地像我的品牌"。所以真正的杠杆是 **design system + 约束**（design tokens、字体/色板硬编码、组件白名单），把模型自由度收窄到"在品牌规范内做变体"。self-review loop 兜下限，design system 锁一致性——这两件事应该串成一句话讲。

### 2.6 截图预览的成本：两个 tool call 怎么优化

这题非常"实战"，是我很喜欢的那种工程取舍题。

**问：** 假设给 agent 一个文件系统，agent 会生成表格写进去。现在加一个预览：让它跑 bash 命令、用 Playwright 截图，再调一个 read 把图片读进来——**这是两个 tool call，中间多一轮循环、多一层成本，怎么解决？**

**答（我）：** 先说为什么非得这样：我让它生成 **HTML**，就是为了能加载出来、用多模态去"看"图片，而不是读 HTML 文本（否则直接用 markdown、read file 就好）。HTML 也更适合 AI 写文档——有富文本，画流程图、放图片视频都强于 markdown。所以这个截图成本我觉得不可避免。随便给两个方案：

1. **定一个新工具 `preview`**：输入表格路径，直接返回一张图片给模型；
2. **让模型在同一个 message 里连发两个 tool call**：一个 bash 截图、紧跟一个 read——因为截图会导出到**固定目录**，路径是定的，read 可以和 bash 写在同一个 message 里。

**问：** 这两种你会选哪个？

**答（我）：** **自己的项目我可能用第一种；生产级、尤其长链路的，我 prefer 第二种。** 理由：bash 是所有 tool 里**成功率最高**的，返回又是文本，AI 理解度最高；而自定义 `preview` 工具更复杂——要多维护 input 硬编码 + output schema，得做 zod / pydantic 校验，越复杂越有概率失败，长链路里一旦出错**追踪到底错在哪一环**会更难。要做 eval test，第二种成功率大概率更高，尤其用便宜模型时。当然如果模型能力起来了、调复杂工具成功率也高，那第一种体验更干净，也可以。

> 💡 **复盘：还有第三种我没提。** 走 **MCP，让工具直接返回 image content block**。MCP 的 tool result 原生支持返回图片，也就是说一个 `render_preview` 工具可以**在一次调用里**既截图、又把图片作为结果直接喂回模型——彻底省掉那次额外的 read，把"两个 tool call"压成"一个"。这是方案一的升级版，同时保留了方案二"少一轮循环"的优点。这道题真正考的是**"成功率 / 可维护性 / token 成本"三者的权衡直觉**——这点我现场答到了，是这场我比较满意的一题。

### 2.7 附件系统设计：参考图 / PDF / CSV / PPT

**问：** 一个设计 agent，用户可能传参考图、PDF（含设计规范）、CSV（每行生成一个设计）、甚至传 PPT 让你改成符合品牌规范、可在我们这边编辑。你会怎么设计这个**附件系统**？

**答（我）：** 先分两类（为简化默认最终产出是 PPT）：

1. **意图分类先行。** 先搞清楚用户传的是"要直接在上面改的"还是"参考图"。要改的，可以类似 Claude Code 的 **PPT skill**，让 agent 直接操作、修改 PPT——因为我希望最终返回的也是**可编辑的 PPT 文档**，而不是像 Nano Banana / 图像模型那样吐一张"图片格式的 PPT"。
2. **建用户级 + 项目级资产库。** 任何产品最宝贵的就是用户数据，这里甚至可以做**长期记忆**：用户偏好赛博风、某种色系，或从素材里发现他总在做某一类东西。
3. **图片相对好办。** 主模型有多模态就直接用；为成本也可以引入一个多模态强的模型——现在还是谷歌最好，国内 Kimi 支持得比较好。CSV / 视频更复杂，视频还得上 FFmpeg 抽帧。
4. **真正的难点在"前期意图确认"。** 用户可能丢一个 CSV（五条成品样式）+ 两张图，但说不清那两张图是五条的统一参考、还是只对应其中一两条、还是只是个形象要重新设计——很容易误导。我在前一段商业研究的项目里也是，研究主题**差一个字就错**（有个真实案例：调研"国内一二线城市、35 岁以上、中高收入、女性、买西式蛋糕的意愿"——一堆定语错一个全错）。所以前期要多轮对话确认意图，再把文件归类：参考的、素材的、配置的（字体文件、配色文件也很常见）。

**问：** 意图识别是靠和用户对话、问问题来做吗？

**答（我）：** 看场景，多轮对话本身有 token 成本。一般分情况：先做一个**检测**——用户第一轮信息是否够开干，设个阈值；或者做一个**复述**让用户 double check。但要注意反面：**意图确认环节太长，用户反而会觉得你们产品能力不行、连我要干嘛都理解不了。** 这是个准确性和体验的平衡。

> 💡 **复盘：这题的"满分答案"长什么样。** 把附件系统拆成四层会更有架构感：**① 接入层**（异构附件解析归一化：PDF 抽文本+版式、PPT 解析成结构化 slide、CSV 解析成行、字体/配色解析成 tokens）**② 分类层**（意图识别，每个附件打标签：参考素材 / 待编辑原件 / 配置规范 / 数据源）**③ 资产层**（用户级+项目级资产库 + 长期记忆）**④ 确认层**（按"信息充分度阈值"决定是否复述/追问，平衡准确性与体验）。把"先分类、再入库、按需确认"讲清楚，比顺着想到哪说到哪更能体现拆解能力。

## 第三轮 · 联创：不聊技术，聊产品观和创业观

二面之后是这一轮——面试官是团队负责人、公司的联合创始人。开场他就说：前两轮技术面已经对我有比较完整的了解，二面那位就是他们的技术负责人（可以理解成 CTO），所以这轮基本不聊技术。结果这成了三轮里我个人收获最大的一轮。

### 3.1 开场：你自己感觉怎么样 + 为什么想做 C 端 / 出海

**问：** 面了两轮，你自己感觉怎么样？

**答（我）：** 二面跟传统面试不太一样——没怎么拷打技术点，更像一次技术交流，我挺喜欢的。我自己选 offer 比较看重公司做的产品，有两个偏好：希望是 **toC**、是**出海**的。出海是因为我是留学生，觉得这方面有优势；C 端是因为我对 toB 的印象是偏无聊、像定制开发。当然有句话叫"C 端做大了就像 B 端"，它们有共通性，但我目前还是想做 C 端。

**问：** 为什么对 C 端更感兴趣？

**答（我）：** B 端我接触到的更像定制开发——接到单了，客户直接提定制需求。而 C 端我要从零去发掘市场、发掘需求，增长阶段也要做很多产品上的思考。还有个刻板印象：C 端在 UI/UX 上要比 B 端打磨得更好——B 端像内部工具更重稳定性；C 端你还需要一个从 0 到 1 的获客渠道，landing page 好不好看会很大程度影响第一印象和后续体验。

### 3.2 聊前一段实习：一个"访谈式大研究"里的子 agent

**问：** 可以介绍下你在前一段实习做的事情吗？

**答（我）：** 那是一家整体做 toB 的公司，我们组做的是一个 toC、垂类出海的 agent，方向是商业研究分析——有点像早期 Gemini / GPT 的 deep research，会去网上搜集大量信息。核心定位是基于**斯坦福小镇**那篇论文，用 AI 去模拟用户画像（我们叫"AI 人设"）。

它的流程是：输入一个研究问题 → 基于 AI 人设去获取社交媒体上的资源、做画像分类 → 然后不只是传统的搜索 + 推理，还会把这些人设当**访谈对象**，默认设定 10 到 20 个目标群体的 AI 人设去访谈。**我做的就是"访谈"这个子 agent**，也能单独拆出来用——有些用户还没想好要不要整份研究报告，只想访谈，就用我这个。

为什么用 AI 来做访谈：① **保证收集质量**——传统问卷里"其他"这种开放项大概率收到低质量信息，我们会判断用户答得准不准、清不清晰，不够好就追问两三轮；如果用户很消极就提前结束，省双方时间。② **AI 拿到了情景上下文**，不像传统问卷一问一答，能在长上下文里挖出更多答案。③ **省时间和钱**——线下找特定群体做访谈本身很贵。

**问：** 你觉得这个过程里比较有挑战的是什么？

**答（我）：** 我做的访谈可能长达半小时甚至一小时，是**长链路问题**，主要挑战在 agent 的稳定性。问题列表可能很多（比如 50 个问题、测下来约 60 分钟），还要给 AI 自由度——不能死板地从第一个问到最后一个，这里有很多 trade-off。为了长链路稳定，也不可能把所有问题全塞进系统提示词，所以设计了一些 tools。这块我也调研了竞品（Listen Lab、ElevenLabs），看他们访谈完的 dashboard 怎么展示数据、一些小交互，再和我们产品融合。

### 3.3 聊更早的初创：迁移、和 CEO 的理念冲突、对赛道的判断

**问：** 可以介绍下更早那段初创实习（fAIshion）吗？

**答（我）：** 那是个很早期的团队，我去的时候主要有两个问题：

1. **上一任技术负责人跑路了。** 项目成立三年，中间迭代很慢、沉淀了一堆历史问题。我们花了大概一个月，借助 AI 把它完整迁移到 **Vercel + Next.js**——因为原本后端拆成了十几、几十个 Python 库，有点离谱。
2. **团队招了很多美国的 unpaid 实习生**，他们对用 AI（写文案、写代码、做增长）都不太熟，所以我也做了不少内部培训和管理的活。

这段我没干很久，主要是和 CEO 在**节奏和优先级上有比较大的理念分歧**，加上我对这个赛道也有点不看好。

**问：** 理念不合，是指赛道看法，还是工作方式？

**答（我）：** 主要是**节奏**。我得承认那位 CEO 在 pitch、招人、给 VC 做 deck、路演上确实很有能力，我当初也是被他说服才加入的。但我感觉他很急、不太愿意沉淀，特别想一直迭代新功能。我提过：迁移已经做完七八成、核心跑通了，哪怕要做 deck 展示都没问题，**我们应该沉淀下来打磨细节**，而不是一味研发新功能。

比如我们 SEO 很烂——产品已经有自然、稳定的用户群，应该把 SEO 做好拿自然流量，而不是一味砸钱靠 Google CPC；landing page 上甚至一直没有 FAQ。因为团队小、又是初创，我没法回避和 CEO 的直接沟通，最后就算了。而且当时这段和前一段实习时间上是重叠的（我同时在干两份），后来想想，不如把精力 focus 在一段上。

> 关于赛道：这个项目技术门槛其实不高，有点像"套壳 API"而不是真正的 Agent 产品。它的优势是有一批稳定的美国用户、迭代有依据（定期发问卷调付费率）。但"云试穿"这个故事我觉得太小了——某电商平台上个月已经上了云试穿，很多中小公司、初创也都在做，甚至往二次元 / 搭配养成方向走，市场竞争很大。所以我比较早就退出了。

**问：** CEO 是觉得 SEO 不重要吗？正常角度，为什么会有人不喜欢免费流量？

**答（我）：** 我觉得他可能更偏"开发思维"。开发是实打实的，一两周就能见效；而 SEO / GEO 这种至少要半个月到一个月才见效、效果还不一定好。所以他的想法可能是：SEO 风险大，不如做新需求；加上人少，他更希望把所有人力 focus 在新需求上。我也能理解，不觉得他就是错的。但合作下来如果我要花精力去争，就有点恶性循环——周会上是全开发团队，不能为这种事无限拉长；于是每周额外开小群，我、CTO、CEO 带着情绪聊，这其实是个恶性循环。

> 💡 **复盘 + 一个提醒。** 这一整段是这场里最"暴露价值观"的部分——面试官其实在通过"你怎么评价上一段合作"看你的判断力和成熟度。我答得算诚实，但**公开发表时要注意分寸**：对前雇主和具体某个人的评价，尽量落在"理念/方法分歧"上，别变成情绪化的差评。这也是我把这段措辞收敛过的原因。

### 3.4 聊个人项目：skill、投毒、和"agent eval 元年"

**问：** 你的个人 AI 项目可以再介绍下吗？挑一个最想分享或最有挑战的。

**答（我）：** 说实话简历那部分没更新、有些过时了。我得坦白：做这些 skill、个人项目、个人网站、开源教程，有一个起因是**我在做自媒体，需要素材去发**——有些开发的活儿就是为了给自媒体提供宣发素材。但里面有些是真有意义的：

- **一个失败的 skill：长文转视频。** 在前公司做的，针对 KOC 人工剪辑成本高——我们有一种内容是用长文对比竞品的 landing page / 博客 / FAQ，很适合转成视频给 KOC 做铺垫（产品上手门槛高，KOC 只按自己理解讲，双方都不满意）。最后失败了：一是 remotion 的 skill 本身比较烂，二是工作量太大（一篇长文几千字）。但这种实践我觉得有意义。
- **一个行业观察：skill 的"通胀"。** 我也在前公司兼职招实习生、做一面。我发现 skill 刚出来时简历写的是"我会用什么 skill"，现在已经变成"我有没有**自己写过 skill、做过 skill 的测评**"。skill 的优点是传播性极强——抖音、小红书随便一个小博主都能发自己的 skill，期末我还看到大学生发自己的复习 skill。但传播性太强导致质量难保证：OpenClaw（小龙虾）刚出来时配套的官方 skill 库 ClawHub，上面 **70% 的 skill 都被投毒**（提示词被恶意注入）；Vercel 的 skills.sh 也是最近才加上很朴素的检测。

这其实映射到我对行业的看法：2025 年是"agent 元年"，大家都在卷 agent 项目；**到今年已经开始做 eval 了**。因为 agent 本质就是"有工具的 LLM"，LLM 输出是概率分布、是黑盒，那 agent 也是黑盒。用户不希望产品像老虎机——抽 10 次中 3 次就算好。所以我自己的个人项目也从"跑出来能用"转向"我要去做测试"。还有一点：这行更新太快了，从 prompt engineering 到 context engineering，再到上周才出来的新名词……所以我觉得**实践大于理念**，我的学习方式就是"干中学"——先干，干到不会了再去补。

> 💡 **复盘：这段我答得挺好。** 它把"个人项目"从"我做了什么"升级成了"我从中看到了行业在往哪走"（skill 通胀 → 投毒 → eval 元年 → 可靠性优先）。**面试里讲个人项目，最高级的讲法不是罗列你做了啥，而是借它讲出你对行业的判断。**

### 3.5 反问①：联创亲口回答"为什么做 AI 版 Canva"

这是我把二面那个没被答透的质疑，原样又抛给联创的一刻。

**我问：** 你们从一个相对传统的工具行业，转去做 AI native 这个新赛道——这个问题我上一轮问过 CTO，但我觉得没被很好地回答到点上。我的质疑是：你们要做 AI 版 Canva，但**你们至少没有设计类的优势**。Canva、Figma 有巨大的用户体量，AI 时代它们也没原地踏步、加了各种 AI 功能；可从 Figma 上市后的股价和舆论看，"Figma + AI"是比较失败的。更 AI native 的像 Lovart，还有 Playground、谷歌 Stitch，偏向不可编辑；Figma、Canva 是可编辑的。CTO 说你们想做"中间"的东西——但"AI + 可编辑"这个中间态已经有一个我觉得挺好的产品 **Pencil**（先出 MCP、后出客户端，我用过）。**所以你们进这个赛道的优势到底是什么？**

**答（联创）：** 这是个很好、也很大的问题，我尽量精简——

> **① 为什么是出海、为什么是现在。** 我们做了十来年工具型产品，很深的一个体会是：在国内做 AI / 工具型产品，尤其从**商业化**角度，非常非常难——前些年被巨头的免费模式带坏了心智，很多人觉得看不见的软件服务不该付费。而海外成熟市场，为好软件付费是很正常的事。所以我们很早就想做能全球化的东西。正好 AI 让出海的难度比以前低了很多，我们的积累也到了，还拿到了海外市场的一笔战略投资，就可以在新领域做尝试。

> **② 为什么是 Canva 这个方向。** Canva 是非常成功的产品，多年积累了海量优质模板（自研 + 高价采购），成了很厉害的护城河，能让小白快速找到模板、改一改就出一个可用设计。但我们的判断是：**Canva 做 AI 其实挺慢、很保守**（大公司船大难掉头、有稳定客户群不敢太激进）；而 Lovart 从第一天就对标"第一个 design agent"，纯 AI native。

> **③ Lovart 和 Canva 面向的是完全不同的用户群。** Lovart 更像给**创意工作者**——需要本身有设计想法的人，和 AI 共创一个 logo 或创意性的东西；这种场景在我们看来其实很小。Canva 更偏**实用性设计**——大部分场景不是创意，而是宣传海报、"我大概要做个什么"，跳模板、用 AI 微调、快速出可用的东西。**即使都在设计领域，定位差异让中间仍有可占的市场。** Canva 哪怕 AI 做得一般，依然有庞大的用户群和收入——这说明它是一个非常大的市场。

> **④ 我们已经在用数据重新定位。** 我们最初上线的是一个**通用型 AI 生图工具**，没特意定位人群，想看谁在用、谁愿意付费。跑了一两个月发现：**超过一半的付费用户在做 marketing 相关的设计**——Instagram 上的多图 carousel、自己开健身房做的 business 海报……所以即使有 Canva 在，依然很多人在找新产品。于是我们正在**重新定位**：从面向所有人的"通用 AI Canva"，收窄到**有宣传需求、有品牌需求的垂直群体**（现在 OPC / 一人公司很火，卖课、做服务都逃不开宣传需求）。

> **⑤ 产品形态 = 品牌 DNA + 自研可编辑器。** onboarding 时引导用户把自己的网站、历史优质设计传上来，让 AI 学他的 **brand DNA**（配色、logo、字体，甚至宣传用语的 tone），之后生成的东西就很像他品牌出来的。第一次可能不满意，所以我们**自研了一套类似 Canva 的编辑器**：图片是**结构化生成**的，哪个字、哪个元素不满意可以双击二次编辑、挪几个 pixel，而不用对着一张图用自然语言反复磨。**每次编辑都变成新信息喂回去**，AI 越用越懂你的品牌——这本身就成了护城河：**一种长期依赖**，只要 business 在跑，就持续有宣传需求。

> **⑥ 关于竞争和护城河。** 也有融了不少钱的创业公司在做几乎一样的定位——但**像的对手不一定是坏事**，可能说明这个需求是一小部分人的共识，就看谁解决得更好。另外 Canva 以前的模板护城河在 AI 时代不一定那么强了——早期很多模板其实也是 AI 生成 + 人工筛，成本很低；**AI 时代真正要担心的，反而是你的产品会不会很容易被 ChatGPT 这种通用 AI 直接取代**（我也可以直接让 ChatGPT 出一张宣传图，可以看看通用 agent Manus 的下场）。但我觉得**垂类还有机会**——不讨论多少年后 AGI 把一切解决，在那之前垂类有很多空间，而且用户也都有 FOMO、在主动找各种 AI 产品。

> 💡 **复盘：这才是"反问"的最高回报。** 同一个质疑，CTO 给的是"中间市场"的框架，联创给的是**带数据、带打法、带护城河逻辑的完整答案**（通用型试水 → 数据发现 marketing 垂类 → 收窄定位 → brand DNA + 自研编辑器 → 长期依赖护城河）。这一前一后让我真切体会到：**好的反问不是为了"考"面试官，而是逼出公司最真实的战略思考——你听完就知道这家公司到底想清楚没有。** 也顺手验证了我在[心态篇](/blog/20260523---agentinterviewmindset/post)里说的："反问是用来筛公司的。"

### 3.6 反问②：现在是不是创业的好时代？

时间快到了（联创说他十几分钟后有个会），我换了个更个人的问题。

**我问：** 您觉得对我们这种**没有创业经历、明年才毕业**的人来说，现在是一个好的创业时代吗？

**答（联创）：** 总体上，是。

> **创业成本比以前低非常多。** 以前老笑话是"我 million dollar 的 idea 都有了，就差一个工程师"；现在有 Claude Code 帮你实现，最后发现你缺的其实是 **idea**、是一个能**持之以恒把执行落地**的人。但这个时代更难的是：**人人都有了做产品的能力**，每天都有海量 AI 产品冒出来，所以"如何让别人知道、获得早期用户、根据反馈迭代、找到商业价值"反而成了更难的事——这也是为什么大家提倡 **building in public**：在 build 的过程中就吸引早期用户、积累影响力，launch 时不用砸广告用户就来了。

> **我的建议：90% 以上的情况，想创业就先加入一家靠谱的创业公司 + 保持自己的 side project。** 靠谱的创业团队会吸纳很多很神奇的厉害的人（比如我们的 CTO，就是个特别厉害、经历很有意思的人——在大厂你一般接触不到这种人、也不会直接汇报给他）；在小团队里你能低成本看到"一个 product 是怎么被 run 起来的"，获得别人要花很多精力才能得到的创业经验，很划算。

> **side project 永远是你自己的。** 创业公司失败率很高，万一黄了你还有自己的事可做、可以再加入别的团队；要是 side project 真的起来了，你就有了更自由的选择。

> **跟更早的时代比，这是更好的产业时代。** ChatGPT 之前，国内互联网其实很固化——该做的创业公司都做完了，大厂格局也稳了。创业需要变量（新技术突破、或社会结构变化），就像 2G→3G→4G 普及才让视频成为可能；**AI 就是当下最大的、质的变量**，所有人都在跃跃欲试。

> 他还顺着我之前一个观点聊了下去——我说过"LLM 火起来是理科生的天下"（更会准确表达、更会 prompt）。他的看法更进一步：**有时候会 coding 的人反而会被经验禁锢**，老想研究"AI 到底哪写得好哪写得不好"，把它变成一个不可解的问题；而文科生压根不在乎底层、改了就完事。他打了个比方：**AI 有点像"新的编译器"**——以前工程师写汇编，有了编译器才用 Python，你写 Python 时不会去纠结编译出来的机器码质量；现在 AI 把人话直接编译成机器能懂的语言，那一点点效率差异可能根本不重要。**所以真正拼的是你对场景需求的理解、以及能不能让别人知道你的产品。** 当然他也补了一句：有工程背景的人依然有优势——更知道 agent 的能力边界、怎么编排工作流、怎么提假设去验证甚至做 A/B 测试。

> 💡 **复盘：这一段我几乎全程在听。** 这就是我在[心态篇](/blog/20260523---agentinterviewmindset/post)里说的——**面试是双向的，把它当成一次能和创始人平等交流的机会，你拿到的东西远不止一个 offer。** 一个面了你三轮的联创，愿意花十几分钟跟你掏心窝子聊创业观，本身就说明这家公司的气质。

## 跨三轮的复盘：我作为候选人学到的

1. **反问的质量，取决于你对赛道做了多少功课。** 我能上来就把"Figma 都被打惨了你们凭什么进场"摆上台面，是因为我真的关注这个赛道——于是面试聊成了同行对话，而不是单方面拷问。把同一个问题在二面、三面各问一次，逼出了 CTO 和联创两种深度的回答，这是我整场最大的收获。
2. **不会的题，承认 + 拆边界，比硬编一个错答案好。** 下拉定位那题我当场承认不知道、愿意搜——没问题；但我接着硬编了一套错的实现，这是最该避免的。说清"难点在溢出裁切、碰撞翻转、堆叠上下文"，本身就是理解的证明。
3. **真实项目经验是最硬的弹药。** 我答得最稳的几题（tool call 成本、附件意图识别、访谈子 agent 的长链路稳定性、design.md 复用），靠的全是真做过的东西，不是背的。这再次印证[修炼手册](/blog/20260309---agentinterview/post)里那句：**Agent 面试题不是要你"准备"，而是你做项目时本来就该想清楚的设计决策。**
4. **越往后的轮次，越不考"你知道多少"，而考"你是个什么样的人"。** 三面几乎不聊技术，聊的是你怎么选赛道、怎么评价过往合作、怎么看行业——这些都藏不住。
5. **被拷打一次，胜过自己看十篇八股。** 我写了五篇"我拷打别人"，这一篇是"我被拷打"。两种视角合起来才完整。

## 写在最后

坐回候选人的椅子，我反而更确认了一直在博客里讲的那几件事：**面试是双向选择**，反问要有攻击性、要带着对赛道的判断去问；**真实做过的项目**远比背下来的八股有力；越到后面，比的越是判断力和价值观，而不是知识点。

这家公司给我留下的印象很好——从 CTO 的技术交流，到联创愿意花十几分钟跟一个还没毕业的候选人聊创业观，都让我觉得这是一支想清楚了自己要做什么的团队。
