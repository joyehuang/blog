## 一、反问环节：我先把赛道质疑甩给了 CTO

我一向主张[反问要有攻击性](/blog/20260528---agentmockinterviewreview/post)，这场我一上来就把最尖锐的赛道问题抛了出去。

**我问：** 你们这个产品 Play 还在内测，是你们从石墨文档这类传统文档业务转型做 AI 产品的一次试水。但文档协作这类产品在如今的市场天花板比较低，你们说想做"AI 版 Canva"。可现在 Canva、Figma 这种设计工具本身就有巨大的用户量，还在不断揭露 AI 功能；更 AI-native 的设计工具出来之后，Figma 无论股价还是舆论都被打得挺惨。**所以——为什么还要进这个赛道？还是你们有不同的看法？**

**答（面试官）：** Figma 和那类纯 AI 工具之间，会有一个**中间市场**。

- **左侧**像 Canva：有海量用户数据、很大的工程团队（可能 2000 多人），追求更低价格、可编辑的设计；
- **右侧**像 Playground、Lovart：价格更高（AI 成本高），但产物**不可编辑**；
- **Figma** 从第一天就面向专业设计师。

而我们的目标用户是**非专业设计师**——大家都有设计需求，但能力满足不了自己。HR 想做招聘海报，得去找内部设计师；运营、个体户、老师、学生，多多少少都有这种需求。我们就是满足这群人的，所以和 Figma 很不一样。

**我追问：** HR 要去找专业设计师，这到底是工具的问题，还是审美（taste）的问题？我自己觉得，现在很多人 vibe coding 抽出来的前端质量差，不是模型的问题，更多是个人 taste 的问题。所以市面上有很多在卖、在营销 taste skill 的；还有些产品就那固定几套模板，我觉得那种活不长久。

**答（面试官）：** 我们还没到 taste 那一步，离它还有点远。HR 找设计资源，更多不是想更好看，而是一个 **0 到 1** 的过程——没有合适的工具，怎么做出一个符合固定要求的海报。更重要的是**品牌一致性**：公司有自己的 logo、设计系统、设计语言，对外宣传的字体长什么样。所以核心是 0 到 1。满足了设计语言之后，你可以做得很好看、也可能不好看，那是第二步——现在第一步都还没有。

**我接：** 为品牌做这件事，感觉和特赞的 toB 业务有点重叠。我第一家实习就在特赞，部门在做 toC 的 agent 产品，但公司主要盈利在 toB——帮大企业管理营销内容、数字资产、媒体，也需要 AI 生产 + 保持品牌一致性。

> 💡 **事后复盘：这段反问为什么"对了"**
>
> 我没有问"贵公司主营业务是什么"这种暴露无知的问题，而是**带着对赛道的判断去挑战创始团队**——直接把"Figma 都被打惨了你们凭什么进场"摆到台面上。这正是我在[心态篇](/blog/20260523---agentinterviewmindset/post)里说的：**反问是用来筛公司的**。一个对自家产品定位说不清的团队，可以直接 pass；而对方能立刻给出"中间市场 / 非专业设计师 / 品牌一致性 / 0→1"这套清晰的差异化叙事，本身就是这家公司值得看的信号。
>
> 这也是为什么我一直强调：**反问环节的质量，取决于你面试前对这个赛道做了多少功课。** 你对赛道越懂，越能把面试聊成两个同行的对话，而不是单方面拷问。

## 二、求职规划：我为什么不读研、不去大厂

**问：** 你还没毕业，对自己有什么样的求职规划？你会看重什么？

**答（我）：** 我本科毕业就想直接就业。家里想让我读硕士，但这个我大概不会听。无论是在特赞做 Atypica、还是做自己的个人项目，我都觉得"做产品"是一件很有意思也很有意义的事——如果我做出一款真正解决了部分人需求的产品，让他们长期用，我就很有成就感。我写技术博客、教程，有粉丝来交流，我也觉得很值。

毕业后我想做的，姑且叫 **Agent 开发**。我对这个词的理解可能和大家不太一样：从 prompt engineering，到 Karpathy 提的 context engineering，再到 agent、harness……名词一直在变，但**产品形态从以前的 SaaS 变成了现在的 agent，只要我一直在做这件事就好**。

更具体一点：我希望成为一个**产品工程师**。我觉得只会写代码的开发，在这个时代确实可能被淘汰。这也是我**抵制去大厂**的原因——大厂里开发更像螺丝钉，你的 KPI 跟 PM 挂钩，需求有没有人用、用的人多不多，都由 PM 兜着，你不需要关心。但在小厂或初创，我得身兼数职，把整个链路跑通：既会开发，又懂用户真正需要什么，还知道怎么用 coding 给用户更好的体验。这才是我觉得有挑战、有意思的地方。

增长也是。很多人 vibe coding 一堆项目，做出来没人用——这是个大问题。从创业团队的角度，花在技术上的精力，**应该小于**花在 GTM、内容运营、广告运营上的精力。把好产品带到目标用户视野里，才是 0 到 1 的关键。我自己也在给个人网站做 SEO/GEO，现在有了 AI，学这些的成本低多了，能快速实践、快速验证想法。

> 💡 **事后复盘：这段我讲得有点散**
>
> 现场我讲了挺长一串，信息密度其实可以更高。如果重来，我会用一句话先收口：**"我想做的是产品工程师——会开发，但更关心需求是否真实、产品如何被用户用起来；所以我选初创而不是大厂，因为初创允许我把'写代码'和'懂用户 + 做增长'这条完整链路跑通。"** 然后再用特赞 Atypica、个人项目、自媒体增长这三个例子去支撑。**先给结论，再给证据**，比顺着想到哪说到哪要稳得多——这其实也是面试答题的通用结构。

## 三、产品协作：我和 CEO 吵过的那一架

**问：** 在你之前的实习里，产品上做的事情有什么印象比较深的？

**答（我）：** 在特赞和另一个初创，我们都不太分前后端，每个人都要想点产品的活。leader 给我的任务常常是**高度抽象**的——比如做海外产品，我得先去 book 竞品的 demo、研究他们的优点和与我们的结合点，做完产品调研才落到开发。

做产品自然分歧会变多，大家都有自己对产品形态的看法，我觉得这是好事。而且我认为在 AI 时代，人和人的差距被相对抹平了，经验没有"AI-native"那么重要，所以更需要交流。

举个具体的：**fAIshion** 的第二个项目是云试穿平台。当时 CEO 要去给 VC 做 pitch、做 deck，很希望核心功能马上上线。但我的想法是——那时我们的 landing page 和 SEO 都做得很烂，不如先把 landing page、SEO 搞好，有了相对稳定的流量再开发核心功能。因为核心功能开发周期很长，而且我们也需要先给已有用户发问卷，确认接下来要做的功能是不是用户真正需要的。那时候就有了争执。

**问：** 做产品会比做技术更容易吵？

**答（我）：** 也不能这么说。准确说是：如果我只是个拿到完整 PRD 就按部就班的开发，那不会吵；但现在我和其他开发都分担了一部分 PM 的活，**自由度高了，自然会激发更多讨论**，不一定是争执。不过这很考验 leader——扁平化、放权的最大难点是大家容易发散，这时候需要一个主心骨把方向带正。开发对产品的理解肯定还是不如专职做产品的人，这得承认；但我觉得现在的开发都需要丰富自己的技能树。

**问：** 技术上和别人有过不同意见吗？怎么解决的？

**答（我）：** 技术上的分歧不多，方案基本要么讨论好了、要么 mentor 直接给。有一个是 fAIshion：接手时项目很乱，拆成好几个服务各自的模块。我们花了不小的精力把它重构到 **Next.js / Vercel** 上——对初创来说上线和使用成本都低，而且产品是出海的。当时还剩几个独立的 Python 后端模块，我的判断是：70%–80% 的功能已经重构完并上线在用了，剩下那部分**能用就先停一下**。我们之所以重构，本来就是因为原后端太乱、老报错，做 pitch、做 deck 都不好看。这其实算不上什么高深的技术问题。

## 四、技术拷打：从零做一个下拉菜单组件

这是整场最硬核的一段，也是我**当场翻车又靠 UX 知识扳回来**的一段。

**问：** 假设让你来做这个产品最初的 UI 系统 / 组件库。比如点一个"……"按钮会弹出下拉菜单——你会怎么做？用什么技术、有什么坑、你怎么思考？

**答（我）：** 前端我基本都用 React，生态比 Vue/TanStack 好太多。这种下拉组件一般都有成熟、可商用的开源组件库。如果只是 build 一个 MVP 或快速上线，很多 SaaS 都用开源组件库——可商用、视觉一致性还高。而且现在不像早期 shadcn 只有五六个主题色，你可以自己配；品牌 logo 是紫色的，调个紫色主题很容易。

下拉菜单这里，关键看你要往里放多少内容。**如果一个下拉超过七八个、十来个选项，不如做成可横向移出的侧边栏。** 因为下拉这种形式是有层级的，会盖住它下方同层级的内容；从用户习惯讲，里面的选项应该比较少，用户在短时间内点开、再点其中一个就走。这里还要考虑二级、三级菜单。

接下来是我专门学 UI/UX 时记住的一个细节——**安全三角（safe triangle）**：

> 如果你用过 GCP 的下拉菜单，会觉得很难用、很别扭，但说不出为什么。其实就是因为它是多级菜单，却没做"安全三角"的阈值判定：你在操作三级、四级菜单时，鼠标稍微一晃出了路径，二级菜单就关了，得重新点。
>
> 安全三角是这样：你点开一级后，**鼠标当前位置会和二级菜单的上下边缘构成一个三角形区域**；只要鼠标还在这个三角形内移动（哪怕是斜着往子菜单挪），就**不关闭**二级菜单；只有移出三角形才隐藏。

**问：** 你一直停在这个三角形里会关闭吗？

**答（我）：** 不会。安全三角就是设一个**缓冲阈值**，让用户在一二级菜单间移动时不会频繁误关子菜单。

**问：** 那如果用户**特别慢**地在三角形范围内移动到菜单，会关吗？

**答（我）：** 这不取决于鼠标移动的快慢，而取决于你**有没有做动态的三角形阈值检测**——只要在三角形范围内就不关。这个我之前复刻过、有动画演示，可以屏幕共享给你看。（最早 Amazon 那个书店导航就有这个。）

到这里都还好。接下来翻车了——

**问：** 这个 dropdown 需要定位——要和 trigger 按钮贴在一起。这个定位你会怎么做？

**答（我）：** 你点按钮，它就在按钮底下出来——这不都是现成组件解决的吗？

**问：** 如果让你**从零实现**呢？

**答（我）：** 🔧 那我确实不知道，愿意搜一下（用 Codex 或 Claude）。大概知道是绝对定位写死在按钮下方，外层父容器写 `relative`，再用 `::after`、加一点 `margin-top` 间隔，水平方向 left 对齐……

> 🔧 **这一段我答错了，下面是正确答案。**
>
> 我把"定位"和"画一个小箭头"混在一起胡诌了（`::after`、`margin-top` 那套根本不是定位浮层的方式）。从零实现一个能贴着 trigger、又不被裁切的下拉浮层，正确的思路链是这样：
>
> **1. 不要只靠 `position: absolute` 写死。** 父容器 `relative` + 子元素 `absolute` 只能解决"贴在按钮下方"的最简单情况，但一旦按钮靠近视口边缘，菜单就会**溢出屏幕**；而且父级只要有 `overflow: hidden` 或 `transform`，菜单就会被裁切或定位错乱。
>
> **2. 用 Portal 把浮层渲染到 `document.body`。** 这样浮层脱离父级的 `overflow` / `z-index` 堆叠上下文，不会被裁切。代价是它和 trigger 不再有 DOM 父子关系，位置得**手动计算**。
>
> **3. 用定位库做计算 —— [Floating UI](https://floating-ui.com/)（Popper.js 的继任者）。** 给它 reference（trigger 按钮）和 floating（菜单）两个元素，它算出坐标，并提供一组 middleware：
>
> - `offset`：菜单和按钮之间的间距（我现场想说的"margin-top"其实就是这个）；
> - `flip`：下方放不下就自动翻到上方（**碰撞检测**）；
> - `shift`：左右快溢出时把菜单推回视口内；
> - `size`：限制最大高度、超出滚动。
>
> **4. 这正是 shadcn / Radix 底层在做的事。** 我面试时说"用现成组件就行"并没有错——[Radix 的 Popper](https://www.radix-ui.com/primitives/docs/components/dropdown-menu) 内部就是 Floating UI + Portal + 碰撞检测。所以更好的答法是：**"成熟组件库之所以好用，正是因为它帮你封装了 Portal + Floating UI 的定位、flip/shift 碰撞检测、focus trap、键盘导航和 ARIA——这些从零写非常容易出 bug，这也是我会优先用组件库的原因。"**
>
> **5. 顺带提一句 2026 年的新解法：原生 CSS。** 现在浏览器有了 [CSS Anchor Positioning](https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_anchor_positioning)（`anchor-name` / `position-anchor` / `position-area`）+ [Popover API](https://developer.mozilla.org/en-US/docs/Web/API/Popover_API)（`popover` 属性 + top-layer 渲染）。它们能把"锚定到 trigger""渲染到顶层不被裁切""自动 flip"这些**不靠 JS 库、纯 CSS/HTML 就做掉一大半**。在面试里能主动提这个，是个明显的加分点。
>
> **复盘心法：被问"从零实现"时，承认不知道 + 愿意搜，没问题（这本来就是我推崇的态度）；但更好的是把它拆成"我知道边界条件是什么"——溢出裁切、碰撞翻转、堆叠上下文——哪怕答不出具体 API，能说清"难点在哪"也证明你真的理解这个问题。** 我当场只做到了前半句，少了后半句。

## 五、Agent 怎么做出"有品味"的设计

**问：** 我现在在做一个类似 Canva 的 agent，你觉得 agent 要怎么设计出**有品味**的设计？

**答（我）：** 市面上一个常见做法是用 **skills**，有很有名的所谓 **taste skill**，类似预制提示词——把有 taste 的人的经验总结、封装进 skill 里。

如果不靠 skill，要让 AI 做出有品味的设计：

- **模型选择很重要。** 有些模型对前端设计有很强的偏好依赖，经典的就是 OpenAI 的 Codex 很爱做卡片式设计。我说的"依赖"是指你想 **zero-shot 抽卡**直接让它抽出来；如果你已经有现成项目、或指令非常精确（改成什么配色、长什么样），各家模型其实都做得不错。难的是"抽卡"。
- **能沉淀成 design system 最好。** 如果目的是大幅贴合公司上下文（品牌调性、字体资产），把这些归纳成一个 design system 非常有价值。我们内部就在做：Atypica 是主项目，基于它做很多子项目，希望用户一看首页就知道"这是 Atypica 的子项目"，从而互相引流——所以我会把主项目的 `design.md` 应用到子项目上。
- **字体这种精确信息，直接硬编码进 markdown。** 我发现一个问题：截一段字体的图去问 AI 这是什么字体，识别率其实很低（不知道为什么）。所以字体名、字号、用了几个 weight、主题色 / 副色 / 字体色 / 二级字体色——这些直接写死在 markdown 里，省心，也不用让 AI 额外烧 token。
- **真要让它有品味，得先让它见过好东西。** 要么去 RL / fine-tune（但"微调出品味"很难）；要么用老办法，喂一批好素材（数量要控制）。
- **让它像 agent 一样自我审核（self-review loop）。** Claude Code 和 Codex 都内置了 preview，可以用无头浏览器截图（Claude Code 用的应该是 Playwright 的 screenshot）。因为很多时候你光看代码（HTML/TSX），只能看出对不对、有没有乱码，**相对位置和颜色只能靠脑补**——颜色在代码里不装插件根本看不出来，不如让 AI 去截图，再用多模态能力看哪里不好、重新试。我自己做过：让 OpenClaw 装上 Playwright screenshot，丢一个简单需求，它自己反复三轮——做、截图、多模态自检、再改——差不多了再把结果交还给我。

**问：** 除了 Claude 和 GPT 系列，你直觉哪个模型比较胜任这个任务？

**答（我）：** **纯抽卡的话，我觉得 Google AI Studio（Gemini）最强**——指的是不靠额外 skill，直接说"帮我做一个什么产品的 demo / landing page"。平时微调一些前端 UI/UX 小样式时，谷歌的 **Flash** 系列对 prompt 的理解也最到位——别的模型可能要写很清楚，Flash 不用。Flash 在这种小修改任务上，性价比、速度、质量我觉得都是最优解。

> 💡 **事后复盘：可以再补一刀**
>
> 这题我答得还行，但少讲了一个对"做产品的 agent"最关键的点：**有品味 ≠ 一次抽卡抽得好，而是"可复现地贴合品牌"。** 对 Canva 这类产品，用户要的不是"惊艳但每次都不一样"，而是"每次都稳定地像我的品牌"。所以工程上真正的杠杆是 **design system + 约束**（design tokens、字体/色板硬编码、组件白名单），把模型的自由度收窄到"在品牌规范内做变体"，而不是放任它天马行空。self-review loop 是用来兜住下限的，design system 是用来锁住上限和一致性的——这两件事我现场是分开讲的，其实应该串成一句话。

## 六、截图预览的成本：两个 tool call 怎么优化

这题非常"实战"，是我很喜欢的那种工程取舍题。

**问：** 假设我们给 agent 一个文件系统，agent 现在会生成表格写进去。现在要加一个预览：让 agent 跑一个 bash 命令、用 Playwright 截图，再让它调一个 read 把图片读进来——**这是两个 tool call，中间多一轮循环、多一层成本，怎么解决？**

**答（我）：** 先说为什么非得这样：因为我让它生成 **HTML**。HTML 可能是更适合 AI 写文档的格式（相比 markdown）——这是之前 Claude Code 一位工程师说的，HTML 有富文本，画流程图、放视频图片的能力都强于 markdown。而我让它生成 HTML，**就是为了能加载出来、用多模态去"看"图片**，而不是去读 HTML 文本——否则我直接用 markdown、read file 就好了。所以这个截图的成本，我觉得是不可避免的。

你的意思是：现在是两个独立 tool call——先 bash 截图、等结果返回，agent 再调 read 读图、再返回，中间两轮循环，即使有 cache 也有额外成本。确实如此。随便给两个方案：

1. **定一个新工具 `preview`**：输入表格路径，直接返回一张图片给模型；
2. **让模型在同一个 message 里连发两个 tool call**：一个 bash 截图、紧跟一个 read——因为截图会导出到一个**固定目录**，路径是定的，所以 read 可以和 bash 写在同一个 message 里。

**问：** 这两种你会选哪个？还是有别的问题？

**答（我）：** **自己的项目我可能用第一种；但生产级、尤其长链路的项目，我 prefer 第二种。** 理由：

- bash 是 AI 调用的所有 tool 里**成功率最高**的，而且 bash 的返回就是文本/代码，AI 理解度也最高（有研究说终端命令是最 AI-native 的）；
- 而你自己设计的 `preview` 工具更复杂：要多维护一层 input 硬编码 + output schema，得做 zod / pydantic 校验，越复杂越有概率失败。在长链路 agent 里，一旦某个复杂工具出错，你**追踪到底错在哪一环**会更难；
- 如果要做 eval test，第二种的成功率大概率更高，尤其用便宜模型时。第二种你也要教模型怎么截图，但那条 bash 命令就一行、固定的，它执行出来终端就给结果；
- 当然这要看模型能力——如果模型能力起来了、调用复杂工具的成功率也高，那第一种（封装好的 `preview`）也完全可以用，体验更干净。

> 💡 **事后复盘：还有第三种我没提**
>
> 现场我给了"自定义 tool"和"同 message 连发"两种，其实还有更优的第三条路：**走 MCP，让工具直接返回 image content block。** MCP 的 tool result 原生支持返回图片（image 类型的 content），也就是说一个 `render_preview` 工具可以**在一次调用里**既执行截图、又把图片作为结果直接喂回模型——彻底省掉那次额外的 read，把"两个 tool call"压成"一个"。这其实是方案一的升级版（一次往返拿到图），同时保留了方案二"少一轮循环"的优点。
>
> 这道题真正考的不是"哪个对"，而是**你有没有"成功率 / 可维护性 / token 成本"三者权衡的工程直觉**——这点我现场答到了，是这场我自己比较满意的一题。

## 七、附件系统设计：参考图 / PDF / CSV / PPT

**问：** 最后一题。一个设计 agent，用户可能传很多附件：参考图、PDF（资料甚至设计规范）、CSV（比如希望每行帮我生成一个设计）、甚至传一个 PPT 让你改成符合品牌规范的、可在我们这边编辑的 PPT。这种场景非常多，你会怎么设计这个**附件系统**？

**答（我）：** 先分两类（为简化，先默认最终统一产出是 PPT）：

1. **意图分类 / 识别先行。** 一开始就要搞清楚：用户传的东西，是"我要直接在它上面改的"，还是"参考图"。
   - 如果是要改的，可以类似 Claude Code 给的 **PPT skill**，赋予 agent 直接操作、修改 PPT 的能力——因为我希望最终返回给用户的也是一个**可编辑的 PPT 文档**，而不是像 Nano Banana / 图像模型那样吐一张"图片格式的 PPT"。
2. **建一个用户级 + 项目级的资产库。** 任何产品最宝贵的就是用户数据。这里甚至可以做**长期记忆**：用户偏好赛博风、偏好某种色系，或者从素材里发现用户总在做某一类东西。
3. **图片相对好办。** 如果主模型本身有多模态能力，直接用主模型；如果为了成本、或发现不同模型各有所长，也可以专门引入一个多模态强的模型——多模态现在还是谷歌最好，国内好像只有 Kimi 支持得比较好。
4. **CSV / 视频更复杂。** 视频如果要支持，还得上 FFmpeg 抽帧，那一套更麻烦。
5. **真正的难点在"前期意图确认"。** 用户可能只丢一个 CSV（里面五条都是想要的成品样式）+ 两张图片，但提示词说得不清楚——那两张图到底是什么？是五条的统一参考，还是只对应其中一两条？还是只是个形象、需要 AI 重新设计？这就很容易误导。
   - 我在特赞做的项目也是这样——商业研究，一份报告产出要 20–30 分钟，研究主题**差一个字就错**。有个真实案例：客户想调研"国内一二线城市、35 岁以上、中高收入、女性、购买西式蛋糕的意愿"——前面一堆定语，错一个全错。设计 agent 同理：这张图到底是什么的图。所以前期要花不少时间，通过多轮对话和用户确认意图，再把所有文件归类：参考的、素材的，还有可能是配置文件（用户丢上来一个字体文件、一个配色文件，这都很可能）。

**问：** 你说的意图识别，是靠和用户对话、问问题来做吗？

**答（我）：** 要看场景，因为额外的多轮对话本身有 token 成本。一般分情况：

- 先做一个**检测**——用户第一轮的信息是否已经达到可以开干的程度，可以设一个阈值/条件，达到就认为"意图识别这轮过了"；
- 或者做一个**复述**：把理解到的任务复述给用户做 double check；
- 但要注意一个反面：**如果你的 double check / 意图识别环节太长，用户反而会觉得你们产品能力不行、连我要干嘛都理解不了。** 所以这里是个体验和准确性的平衡。

> 💡 **事后复盘：这题的"满分答案"长什么样**
>
> 我现场答得比较全了，但如果要更结构化，附件系统其实可以拆成清晰的四层，面试时这样讲会更有"架构感"：
>
> 1. **接入层**：统一把异构附件（图/PDF/CSV/PPT/字体/配色）做解析与归一化——PDF 抽文本+版式、PPT 解析成结构化 slide、CSV 解析成行、字体/配色解析成 tokens；
> 2. **分类层**：意图识别——每个附件打标签（`参考素材` / `待编辑原件` / `配置/规范` / `数据源`），这一步决定后面怎么用它；
> 3. **资产层**：用户级 + 项目级资产库 + 长期记忆（偏好沉淀、去重、跨项目复用）；
> 4. **确认层**：基于"信息充分度阈值"决定是否要复述/追问，平衡准确性与体验。
>
> 把"先分类、再入库、按需确认"这条主线讲清楚，比顺着想到哪说到哪更能体现你对复杂系统的拆解能力。**这也是我整场最该补强表达结构的一题。**

## 写在最后（作为候选人，我的几点感受）

坐回候选人的椅子，我反而更确认了自己一直在博客里讲的那几件事：

1. **反问的质量，取决于你对赛道做了多少功课。** 我能上来就把"Figma 都被打惨了你们凭什么进场"摆上台面，是因为我真的关注这个赛道——于是面试聊成了两个同行的对话，而不是单方面拷问。这就是[心态篇](/blog/20260523---agentinterviewmindset/post)里说的"聊到同一张桌子上"。

2. **不会的题，承认 + 拆边界，比硬编一个错答案好。** 下拉定位那题，我当场承认不知道、愿意搜——这没问题；但我犯的错是**接着硬编了一套错的实现**。更好的做法是：哪怕答不出具体 API，也要能说清"难点在溢出裁切、碰撞翻转、堆叠上下文"——**说清楚问题边界，本身就是理解的证明。**

3. **真实项目经验是最硬的弹药。** 这场我答得最稳的几题（tool call 成本、附件意图识别、design.md 复用），靠的全是特赞 Atypica、fAIshion、OpenClaw 这些真做过的东西——不是背的。这再次印证了我在[修炼手册](/blog/20260309---agentinterview/post)里的那句话：**Agent 面试题不是要你"准备"，而是你做项目时本来就该想清楚的设计决策。**

4. **被拷打一次，胜过自己看十篇八股。** 我写了五篇"我拷打别人"，这一篇是"我被拷打"。两种视角合起来才完整——**面试官席让你知道"什么样的答案站得住"，候选人席让你知道"被追到墙角是什么感觉"。**
