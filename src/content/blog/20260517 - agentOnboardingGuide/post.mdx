## 第一章｜认识 AI Agent

> 在你能"开发 Agent"之前，先要能"看懂 Agent"。这一章帮你三句话讲清楚 Agent 是什么、它和已经知道的那些东西（ChatGPT、API、聊天机器人）的区别在哪。

### 1.1 一个不严谨但好理解的定义

> **如果说大语言模型（LLM）是一颗"大脑"，那么 Agent 就是这颗大脑加上了眼睛、手、记忆，以及一个能让它自己循环工作的身体。**

更严谨一点的说法：**Agent 是一个以 LLM 为决策核心、能够感知环境、调用工具、维护记忆、并通过自主循环完成多步任务的系统。**关键词四个：**LLM、感知、工具、循环**。

### 1.2 Agent 不是什么：三个常见误解

**误解一：Agent ≠ 聊天机器人。** 聊天机器人的核心是"对话"；Agent 的核心是"完成任务"——对话只是它接收任务的一种方式。

**误解二：Agent ≠ 单次 LLM API 调用。** 单轮调用是"输入 → 输出"；Agent 至少要具备其中一项才能算 Agent：**循环、工具、状态**。

**误解三：Agent ≠ 一个用 LLM 包装的表单 App。** 把 LLM 塞进一个用户填表单的产品里——和"加了 GPT-4 的搜索框"没本质区别，因为模型没有参与任何"下一步做什么"的决策。

但这里要马上修正一个 v1 版本里我讲得太满的地方（也是很多入门材料的通病）：**"模型自己决定每一步"不是 Agent 的资格考试——自主程度是一条光谱。** 真实生产系统里最常见的形态，恰恰是"确定性工作流控制主干，LLM 只在分类、规划、工具选择这些局部节点做决策"——权限、重试、审批、状态转换都由程序控制。刻意**减少**模型的自由度，往往是可靠性工程的正解，而不是"伪 Agent"。Anthropic 官方的建议也是把 workflow 和 agent 放在同一条光谱上按需取用。所以更准确的说法是：表单 App 的问题不是"不够自主"，而是模型没参与任何决策；**一个好系统该给模型多少自主权，是工程取舍，不是身份认证。**

需要补充一点：回到 2023 年的早期 ChatGPT，给它一个输入它给你一个输出，仅此而已。到了 2025–2026 年，主流 ChatGPT、Claude、Gemini 网页端已经把不少 Agent 化能力内置进了产品——联网搜索、代码执行、文件读写、MCP 工具调用。**但这是"厂商在产品层把 Agent 能力打包好给用户用了"，而不是"LLM 本身变成了 Agent"。** 底层模型仍然是那颗大脑，能动手是因为外面套了一整套 Agent 工程——而你要学的，就是这一整套工程。

### 1.3 Agent 的四件套，以及把它们装起来的 Harness

理解这四件套，你就理解了 Agent 80% 的工程实质。

**大脑（LLM）**：思考和决策。接收当前状态，输出下一步动作（回答用户 / 调用工具）。

**记忆（Memory）**：先按两层建立直觉——**短期记忆**是当前任务的上下文（用户刚说了什么、刚调了什么工具）；**长期记忆**是跨会话的持久信息（用户偏好、关键事实）。工程实现里其实再细分为 Working / Short-term / Long-term 三层，第五章会展开。

**手（Tools / Function Calling）**：Agent 与外部世界交互的接口——搜索引擎、代码执行、邮件 API、数据库查询都是工具。主流 LLM 厂商都提供原生的 Function Calling 能力。

**循环（Agent Loop）**：`感知 → 思考 → 行动 → 感知 → ...` 最经典的实现叫 **ReAct**（Reason + Act）。终止条件通常是：模型自己判断完成 / 达到最大步数 / 用户中断。

而把这四件套**组装起来**的那层工程——决定模型每一步"看到什么、能动什么、错了怎么办"的全部脚手架——行业在 2026 年给了它一个正式的名字：**Harness（可以理解为"驾驭层"）**。LangChain 负责开源 agent 的工程师 Vivek Trivedy 有句话很好记：**"Agent = 模型 + Harness。模型之外的一切，都是 harness。"**

这不是一句空话。LangChain 在 2026 年 2 月[做过一个实验](https://www.langchain.com/blog/improving-deep-agents-with-harness-engineering)：模型一行不动（用的还是闭源的 GPT-5.2-Codex），只重做 harness——系统提示词、工具、防死循环的中间件——就把自家 agent 在 Terminal-Bench 2.0 上的成绩从 52.8% 拉到 66.5%（+13.7 分，当时冲进榜单前五）。**很多看起来是"模型不行"的问题，其实是 harness 没做好。** 你在这份文档后面要学的所有东西——上下文、记忆、工具、可靠性——本质上都是在学怎么造好一个 harness。

### 1.4 一个最小 Agent 工作流：以"帮我订午饭"为例

你对一个 Agent 说"帮我订一份午饭，30 块以内，最好是粤菜"。它内部大致这样跑：

**第一轮**：思考"先查附近粤菜店" → 调用 `search_restaurants(cuisine="粤菜")` → 拿到 8 家店。

**第二轮**：思考"按预算过滤" → 调用 `filter_by_price(max_price=30)` → 剩下 3 家。

**第三轮**：思考"信息够了，给用户" → 直接回答："我找到了 3 家，A、B、C，您选哪家？"

注意四个关键点：

1. **每一步都是模型自己决定的**——它自己判断要不要调工具、调哪个、什么时候停。这是 Agent 和"硬编码工作流"最核心的区别（当然，按 1.2 说的，真实系统里这个自由度是被刻意设计过的）。
2. **工具执行不是模型在做**——模型只是输出"我想调这个工具"，真正执行是你程序的工作。
3. **记忆贯穿了整个流程**——第三轮思考时，模型还记得最初的预算和口味要求。
4. **有真实副作用的动作要过"程序层"的关**——真到下单付款那一步，负责任的 Agent 一定会先向你确认；而"能不能下单、金额上限多少"这类边界，是程序在工具执行层强制的，不靠模型自觉。这一点非常重要，第五章会专门展开。

### 1.5 本章一句话

> **Agent = 一个 LLM 在一个循环里，反复地"想一下、做一下"，直到任务完成——而这个循环周围的一切工程，叫 harness。**

**"谁最强"这个问题，2026 年年中第一次变得真正胶着**：官方 [Terminal-Bench 2.1 榜单](https://www.tbench.ai/leaderboard/terminal-bench/2.1)上（截至 2026-07-11），Codex CLI + GPT-5.5（83.4%）和 Claude Code + Fable 5（83.1%）是统计学平手，榜首每几周换一次人。所以 v1 那句"做 Coding Agent，Claude 是默认选择"要降级成：Claude 在 Agent 工程生态上仍是口碑默认，但**别把选型判断外包给任何静态结论，看最新榜单**。

**关于选型的四条真实建议（这部分不过期）**：

1. **不是越大越好，按任务分级使用**——简单意图识别、路由用便宜小模型；复杂代码生成、推理才用旗舰。这是控制成本的关键技巧。
2. 大部分国内模型都提供 **OpenAI 兼容 API 端点**——同一套代码改 `base_url` 就能切换模型。
3. **不要押宝单一模型**——成熟项目会同时接 2–3 家，按任务路由。
4. **模型选型的权重正在下降。** 还记得 1.3 那个实验吗：模型不动，只改 harness，+13.7 分。同档位模型之间的差距，常常小于 harness 质量的差距——把纠结型号的时间省下来，花在工具设计和上下文工程上，回报更高。

### 2.2 框架 / Harness 层：用什么搭脚手架

主流选择分两类：**官方 SDK / Harness** 和 **第三方框架**。

**官方阵营：**

- **OpenAI SDK**：最古老、最稳定。**但只能调 OpenAI 自己的模型**——不过因为 OpenAI 兼容协议是事实标准，DeepSeek / Qwen / Kimi / GLM 这些国内模型都可以通过改 `base_url` 用它调。
- **Anthropic SDK**：Claude 官方 SDK，Tool Use、Computer Use 这些原生能力都用它接最顺。
- **Claude Agent SDK**（2025 年 9 月由 Claude Code SDK 更名而来）：注意它和上面那个不是一回事——它不是"调模型的 SDK"，而是"**给 agent 一台电脑**"的完整 harness：内置文件系统、终端、权限系统，和"收集上下文 → 行动 → 验证"的循环。想直接站在成熟 harness 上做产品，这是当前最顺的入口之一。
- **Google Gen AI SDK**：Gemini 官方 SDK，多模态接入最直接。

**第三方框架（按推荐度排序）：**

**Vercel AI SDK**（我个人最推荐的入门选择，这个判断从 v1 保持不变，而且被加强了）：

- **不限模型**——`import { anthropic } from '@ai-sdk/anthropic'` 用 Claude，换成 OpenAI / Google / DeepSeek 只改一个 import，其他代码完全不动。
- **抽象轻**——给你的是"流式输出""工具调用""结构化输出"这些原语级别的能力，不会像 LangChain / LangGraph 那样强制你按它的抽象（Chain / Graph / Node）来组织代码。Agent Loop 怎么编排完全由你决定。
- 2026 年 6 月发布的 AI SDK 7 里，Agent 已是一等公民：ToolLoopAgent（工具循环）、WorkflowAgent（可中断恢复的长任务）、还有 **HarnessAgent——把 Claude Code、Codex、Deep Agents、OpenCode 这些现成 harness 包成同一个接口，像换模型一样换 harness**。（注意 v7 要求 Node 22 + ESM。）
- TypeScript / Next.js 生态尤其顺。

**LangChain / LangGraph**：最老牌、生态最大、文档最丰富，但也最常被吐槽"抽象太重"——这个吐槽 2026 年依然成立。值得注意的是 LangChain 自己的重心也明显转向了 **deepagents**，官方口号就是"batteries-included agent harness"。除非你的项目特别需要 LangChain 的现成组件，否则**不建议作为第一站**。

**Pydantic AI**：对标 Vercel AI SDK 的 Python 侧轻量选择，2026 年 6 月的 V2 版设计哲学明写"harness-first"。**CrewAI / AutoGen / Mastra**：分别针对多 Agent 协作、企业级编排、TypeScript 全栈，有具体场景再选。

**给入门者的建议**——现在有两条被验证的路径，而且不冲突：

- **路径一（v1 就有）**：第一个 Agent 用 **Vercel AI SDK 或厂商官方 SDK** 手写——抽象足够轻，不会挡住你看清"Agent Loop 到底在做什么"。
- **路径二（2026 年被反复验证）**：**从第一天起深用一个现成 harness（Claude Code 或 Codex），把它当活教材**——观察它怎么组织上下文、怎么设计权限、什么时候压缩历史。Philipp Schmid 和 Addy Osmani 都在 2026 年上半年专文论证过"新人该从 harness 起步"。Osmani 那句话值得记住：**"一个普通模型配一个好 harness，胜过一个顶级模型配一个烂 harness。"**

**不要为了"学框架"而学框架**——框架本身不是简历资产，"我用某个框架做了什么有价值的事"才是。

### 2.3 协议层：MCP 与 Skills

**MCP（Model Context Protocol，模型上下文协议）** 是 Anthropic 2024 年底推出的开放协议，让"LLM 应用"和"工具/数据源"之间有了**标准化的对话方式**。

类比一下——以前每个 Agent 想接一个新工具都得自己写适配代码；MCP 就像 Agent 领域的"USB 标准"。这里 v2 有个重要更新：**2025 年 12 月，Anthropic 把 MCP 捐给了 Linux Foundation 旗下新成立的 Agentic AI Foundation（和 OpenAI、Block 共同发起）**——它从"某家公司的协议"变成了中立基金会治理的行业标准，官方注册表已收录约一万个 MCP Server。另一个时间点值得记：MCP 史上最大的一次版本修订（无状态化改造）定档 **2026-07-28** 发布——你读到这里的时候，它可能刚刚定稿。

入门建议不变：先**用别人写好的 MCP Server**，把 Notion / GitHub / Slack 这些工具直接接到你的 Agent 上。

**Skills 是同样需要重点理解的概念。** 它是 Anthropic 2025 年 10 月推出的**产品形态**——把一个"打包好的、可复用的 Agent 能力模块"作为一等公民。v1 写的是"业界正在跟进"，现在可以改成完成时了：**2025 年 12 月起 Skills 成为开放标准（[agentskills.io](https://agentskills.io)），到 2026 年 7 月已有 40+ 客户端采纳**——包括 OpenAI Codex、Gemini CLI、GitHub Copilot、VS Code、Cursor。

要理解 Skills，先看一个工程现实：一个真正能用的 Agent，往往不只是"模型 + 几个工具"，它还需要——一组**专门的工具**（生成 PPT 需要 pptx 库）、一份**精心调过的 Prompt 指导**（什么时候用什么、有哪些坑）、一些**示例和参考资料**（让模型知道"好的输出长什么样"）、有时候还需要**特定的代码片段**。

如果这些散落在 Prompt、工具描述、代码注释里，会发生两件事：模型不知道什么时候该用什么；这些能力没法被复用、分发、版本化。

**Skills 把这套"能力包"变成一个可独立存在、可加载、可分享的产品形态。** 一个 Skill 通常是一个文件夹，里面有 `SKILL.md`（描述这个 Skill 做什么、什么时候触发）+ 相关脚本、工具、参考资料。Agent 在需要的时候自动加载对应的 Skill。

**几个关键直觉：**

- Skills 是给 **Agent 用的"使用说明书"**，不是给人看的文档——语言是写给模型读的。
- Skills 解决了"无限工具会让模型混乱"的问题。Agent 接 50 个工具，模型选错的概率极高；但如果分组装进 10 个 Skills，模型只在匹配到这个 Skill 的场景才会看到对应的工具——这是**按需暴露**。
- **Skills 和 MCP 互补**：MCP 解决"工具如何被调用"（接口标准化）；Skills 解决"工具如何被组织和触发"（能力打包）。一个 Skill 内部可以调用多个 MCP Server。

**三者的关系**：

- **Function Calling / Tool Use** 其实指的是同一回事——OpenAI 叫 Function Calling，Anthropic 叫 Tool Use，本质都是"LLM 能不能输出'我想调这个工具'的指令"这个底层能力。
- **Skill** 是另一层抽象——把一组工具 + Prompt 指导 + 参考资料打包成一个可复用、可加载的能力模块。

顺便记两个名字，看 JD 时不至于陌生：**AGENTS.md**（仓库级的 agent 指南文件约定，已有 6 万+ 开源仓库在用）和 **A2A**（agent 之间的通信协议，150+ 组织支持）。

### 2.4 数据层：RAG、Agentic Search、Memory

这一层在 2026 年变化最快——传统 RAG 范式正在被几种新形态部分替代。

**RAG（Retrieval-Augmented Generation）** 是 2023–2024 年最主流的"让 LLM 拥有自有知识"的方式。核心组件是**向量数据库**（Milvus、ChromaDB、Pinecone、pgvector、sqlite-vec）。

回到 RAG 出现的时代（2023 年初），主流 LLM 的上下文窗口只有 4K–32K tokens，长文档塞不进去，只能"切片 + 向量检索 + 拼回 prompt"。今天主流旗舰模型已经普遍支持 1M token——对很多场景来说，"先检索再生成"的复杂管道根本没必要，直接把文档塞进上下文里效果反而更好。

到 2026 年年中，"RAG 死没死"这场争论基本收敛成一句话：**朴素的"切片 + 向量检索"不再是默认架构，但"检索"本身作为 Agent 的一个工具动作活得很好。** 新范式叫 **Agentic Search（agent-as-retriever）**：不建索引，让 agent 用 grep / 关键词按需探索。两个标志性证据：Claude Code 至今不给代码库建任何向量索引，靠文件系统工具现场找；AWS 发在 AAAI 2026 的论文标题干脆叫[《Keyword search is all you need》](https://arxiv.org/abs/2602.23368)——纯关键词的 agentic 检索能达到 RAG 90% 以上的效果。但反方证据也要给你：[Cursor 的官方实验](https://cursor.com/blog/semsearch)显示，在 grep 之上**叠加**语义检索，仍能平均再提 12.5% 的回答准确率。所以教给你的结论是：**grep / agentic search 优先，规模和召回要求上来了再加 embeddings——两者是叠加关系，不是替代关系。**

**LLM Wiki** 是这个方向上另一个值得知道的模式：Karpathy 在 2026 年 **4 月初**（v1 误写作 5 月，顺手纠错）发布了[这个想法](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)——对个人/小团队的中等规模知识库，放弃向量检索，把所有内容用 Markdown 组织成一个"维基百科"结构，由 LLM 增量维护、按需塞进上下文。优点：没有切片误差、没有召回率问题、调试极简单、还能命中 Prefix Cache 省钱。不过两个月后的第一波长期实测给出了比较清醒的结论：对窄场景有效，但维护成本大约等于省下的时间。**别神化，当一个值得亲手试的模式即可**（第四章会让你亲手对比）。

**但传统 RAG 没死。** 它在以下场景仍然首选：

- **企业内部场景，强调数据隐私**——文档不能离开企业网络。
- **超大规模知识库**（百万级文档以上）——上下文塞不下、grep 也扫不动。
- **多用户、多租户场景**——每个用户数据要隔离。

简单说：**做个人项目，先试 agentic search / LLM Wiki；做 toB 企业项目，RAG 仍是默认。**

**Memory（记忆系统）** 和 RAG 有重叠但不同——Memory 更强调"对当前用户/会话的个性化记忆"，通常分三层：Working / Short-term / Long-term，第五章专门讲。2026 年它已经从概念变成了产品层：Anthropic 的 memory tool 正式 GA（注意它是**文件式**的——agent 在一个 /memories 目录里读写 Markdown，不是黑盒向量库）；OpenAI 在 6 月重写了 ChatGPT 的记忆系统（"Dreaming"，记忆会随时间自我更新）；开源侧 Mem0 / Zep / Letta 在记忆基准上互卷。**一个值得注意的共同点：实践里赢的记忆方案，大多是"文件式 + agent 自己管理"**——memory tool、CLAUDE.md、Karpathy 的 wiki，全是这个形态。

### 2.5 应用层：主流落地场景，和 2026 年中的三个新动向

经过一年多的行业观察，被反复提及、跑通商业化的主要是这三个方向：

**AI 搜索**：从 Perplexity 到秘塔 AI 到 Felo，本质都是 Agent 化的搜索——不再是"关键词匹配+排序"，而是"理解意图+检索+综合+生成"。这个方向已经证明能收钱：Perplexity 的年化收入 2026 年 3 月已到约 4.5 亿美元。

**Chat-to-BI**：让业务人员用自然语言查数据、生成图表、做归因分析。国内也开始"能收钱"了——阿里云 DataWorks ChatBI 从 2026 年 4 月起正式商业计费。

**Vibe Coding**：从 Copilot 到 Cursor 到 Claude Code——这个方向最近一年进展最快，也最直接地颠覆了传统软件开发。两个数字感受一下量级：Claude Code 的年化收入 2 月已超过 25 亿美元；OpenAI Codex 的周活 6 月突破 500 万，**其中约 20% 是非开发者**。

在这三个方向之外，2026 年年中有三个新动向，值得你在选方向时知道：

**一，Agent 出圈了。** Claude Cowork（面向非技术工作者的桌面 agent）的官方数据显示 90% 以上的会话是非编程工作；OpenClaw（把个人 agent 接进 WhatsApp / Telegram / 微信的开源网关）在 GitHub 冲到 38 万 star，是史上最快的 star 曲线之一。"Agent = 程序员玩具"的阶段结束了。

**二，通用 vs 垂类，天平在向垂类倾斜。** 通用 agent 是模型厂和平台的战场——Meta 收购通用 agent 明星 Manus 的交易在 4 月被监管否决、6 月正式告吹，Manus 自己也在转垂类 task packs；而垂类在赚真钱：法律 agent Harvey 年化收入约 3 亿美元，客服 agent Sierra 破 1 亿美元。这个逻辑的行话叫 **context moat**：领域上下文 + 工作流 + 结果交付，是模型更新卷不走的部分。普通开发者的机会在垂类——这个话题我在[第二场粉丝分享](https://www.joyehuang.me/talks/2026-ai-product-form.html)里展开讲过。

**三，产品形态在换代。** Karpathy 6 月给了一个三段论：LLM 产品经历了"你去访问的网页（ChatGPT）→ 装在你电脑上的 app（Claude Code / Cursor）→ **住在组织里的常驻执行体**"。第三段的代表是 Anthropic 6 月底发布的 Claude Tag：一个常驻 Slack 频道、有记忆、能接异步任务、整个团队共用的 Claude。对你的意义：**"Agent 工程师"的工作对象，正在从"一个聊天窗口"变成"一个数字同事"。**

## 第三章｜怎么看这个方向：趋势、心态与避雷

> 这一章不讲技术。讲三件事：为什么 Agent 是当下最值得入门的方向、为什么你不需要焦虑、以及怎么避开那些看上去很美但其实是坑的机会。

### 3.1 "晚不晚"是一个伪命题

咨询中我被问得最多的一个问题是："Joye，我现在才开始学是不是太晚了？"

我的标准回答是：**这条赛道上"老人"也才两年经验。**

简单梳理时间线：

- **2022 年 11 月** ChatGPT 发布
- **2023 年初** AutoGPT 等开源项目兴起，"Agent"开始被广泛讨论
- **2023 年 6 月** OpenAI 推出 Function Calling，Agent 工程化进入新阶段
- **2024 年** Cursor 进入商业化爆发期
- **2024 年底** Anthropic 推出 MCP，Agent 协议层开始成形
- **2025 年** Manus、Claude Code、Devin 等 Agent 产品集中爆发
- **2025 年 10–12 月** Anthropic 发布 Agent Skills；MCP 捐入 Linux Foundation；Skills 成为开放标准
- **2026 年上半年** "Harness Engineering"成为行业词（Anthropic 工程博客系列、LangChain 的 +13.7 分实验、Osmani 的命名文）
- **2026 年 6 月** "Loop Engineering"出现——Steinberger 一条推引爆，Osmani 当天系统化；同月 Claude Tag 发布，Agent 进入"组织常驻"形态
- **2026 年 7 月** Lilian Weng 发表 harness 工程综述——这个领域开始有"教科书"了

也就是说——**目前行业里所谓的"资深 Agent 工程师"，从入行到现在最多两到三年。** 换一个更扎心的说法：**没有人有"十年 Loop Engineering 经验"——这个词今年 6 月才出现，最资深的工程师和你读的是同一条推、同一篇博客。** 资历会过期，好奇会复利。和传统开发领域那些有十年、二十年经验的方向比，这是一个真正可以**靠学习速度追赶上来**的方向。

### 3.2 为什么是现在：三个判断

**判断一：Agent 正在从"Demo 期"走向"生产期"。**

2023–2024 上半年，行业里大量 Agent 项目停留在 Demo 阶段。从 2024 年下半年开始，可靠性、可观测性、Eval 体系、成本控制这些"工业级"问题被认真对待——**这是一个工程师真正能发挥价值的阶段。**

**判断二：基础设施层的标准化窗口期。**

MCP、Skills、AI Gateway 这些基础设施层的标准还在快速成形（MCP 最大的一次修订这个月底才定稿）。这意味着如果你现在入场，**有机会真正参与到一个新行业基础设施的建设里**——这种窗口期在传统开发领域非常稀缺。

**判断三：人才需求正在系统性增长。**

这个判断在 2026 年有了硬数据，海内外都有：

- **Y Combinator**：W26 批次（2026 冬）约六成是 AI 公司、超四成做 Agent 基础设施；到 6 月刚结束 Demo Day 的 **P26 批次（2026 春），做 LLM Agent 的公司已占约七成（137/196）**。当全世界最挑剔的孵化器把七成筹码押在 Agent 上，这个赛道的人才需求只会继续涨。
- **国内校招**：**阿里 2027 届校招（2026 年 3 月启动）AI 相关岗位占比超 80%，并首次新设"AI Agent 优化工程师""Agent Infra 工程师"等七类 AI 岗位**——"Agent 工程师"第一次字面进入大厂校招职级体系；字节 Seed 的 2027 届基础模型校招 4 月开启。脉脉春招报告（2026 年 5 月）：AI 岗位量同比增长 8.7 倍。
- **海外数据**：斯坦福 AI Index 2026 首次把"Agentic AI"列为独立技能簇，岗位提及量年增 280%+；LinkedIn 年度增长最快职位榜第一名连续第二年是 AI Engineer。

### 3.3 你不需要焦虑的几个理由

**第一，行业没有"绝对权威"。** 传统计算机科学有那种"我看过他的论文""我读过他的书"的权威专家。Agent 这个方向太新，没有这种人物。OpenAI、Anthropic 的最佳实践都是工程师在边做边写——他们和你的差距是"实践积累"，不是"天赋差距"。

**第二，信息差极小。** OpenAI 和 Anthropic 把关于 Prompt 工程、Agent 设计、Skills 体系的最佳实践直接发在公司博客上，**任何人免费可读**。这种透明度在传统行业不可想象——你想学的东西大多都已经被写出来了，问题只是你愿不愿意花时间读。

**第三，工具门槛的"简单部分"在下降，但"深的部分"在上升。** 这一点需要分两半讲。

往简单方向看：三年前你想接入 LLM 要懂 GPU 部署；今天你只需要会调 API。Vibe Coding 让"做一个个人网站""做一个简单 ChatBot"几乎没门槛。

但这恰好意味着——**当造一个"没有壁垒的 Agent 项目"变得很简单时，没有壁垒的项目本身就不值钱了。** 你在 GitHub 上能找到一千个"AI 健康助手""AI 客服 Bot"，因为每个人都能用 Vibe Coding 在一个周末造一个。这些项目放在简历上，面试官扫一眼就知道含金量。

真正的入门门槛被推到了**更深的位置**：你能不能选一个**真实存在的、不是用现成 ChatGPT 就能解决**的问题做项目？你能不能在工程层面做出**像样的取舍**？你能不能讲清楚每一个技术选型"为什么是它"？

简单说：这不是"门槛下降"，而是"门槛从代码能力转移到思考深度"。我后来把这个判断进一步压缩成一对词：**知识是 content，AI 能无限供给；你的判断、品味、踩过的坑是 context，这才是护城河**——展开版在[《从实习面试到 Agent 开发》](https://www.joyehuang.me/blog/20260615---frominternshipinterviewstoagentengineering/post)那篇里。

**第四，对手不是别人，是去年没动手的你自己。** Agent 领域真正"卷"的，不是知识储备，而是"动手量"。读 100 篇博客不如自己写一个能跑的小 Agent。**哪怕你今天才开始，只要你动手，你就在跑赢"只看不动手"的 90% 的人。**

### 3.4 七个常见误区

这些误区是我接咨询时遇到最频繁的，逐条破除——

**误区一："我数学不好，搞不了 AI。"** 你想做的是 **AI 应用**，不是 **AI 算法**。应用层的工程实践基本用不到数学。

**误区二："要先把 LLM 原理学完才能学 Agent。"** 反了。应用层和底层算法是**两条相对独立的赛道**。先做应用，等遇到具体问题再回头补理论，效率高 10 倍。

**误区三："学这个是不是要会很多框架？"** 框架是工具不是目标。理解了本质你看任何框架都是几分钟上手；只会用框架但不懂底层，遇到框架不支持的场景就完全束手无策。

**误区四："我没大厂背景就找不到工作。"** Agent 这个方向恰好是**创业公司的天下**。他们看重的是"你能不能立刻干活"——你的项目经验、GitHub、博客比学校和上家公司更重要。

**误区五："AI 发展这么快，学了会不会马上过时？"** 变的是表层名词，不变的是底层对象。**Agent Loop（ReAct 只是它 2022 年的名字）、上下文工程、记忆系统、工具调用、Eval 体系**——名词一直在换（Prompt → Context → Harness → Loop 的接力就是活例子），但这些工程对象过去几年没变过，未来五年也不会有本质变化。

**误区六："Agent 是不是已经是红海了？"** 真正"卷"的领域是已经被深度开发了二十年的传统领域。Agent 在 Coding、Research、Customer Support、BI、Marketing 等领域的渗透率还不到 10%——这才哪到哪。

**误区七："Multi-Agent 就是堆更多 Agent，越多越强。"**（v2 新增）行业在 2025 年为这个想法交过学费——Cognition 那篇著名的《Don't Build Multi-Agents》就是当时的反思。到 2026 年，共识回摆成一个清晰的范式：**Multi-Agent 有效，但形态是"一个主控 agent + 隔离的子 agent + 写操作单线程 + 子 agent 只交回摘要"**——Cognition 自己在 4 月发文修订了立场；Anthropic 用 16 个并行 Claude 写出 10 万行的 C 编译器，用的正是这个范式。对新人的意义：先把单 Agent 做可靠；Multi-Agent 是"编排问题"，不是"数量问题"。

## 第四章｜怎么入门和准备求职

> 这是这份文档最实操的一章。从你"打开 IDE 写第一行 LLM 调用"开始，到"拿到第一个 offer"为止，需要做的事都在这里。v2.0 给每个阶段加上了前置条件、最小交付物、验收标准和预算——让"我到底学到哪了"这件事可以被你自己检验。

### 4.1 前置技能：你需要会什么、可以不会什么

**先说语言。** Agent 开发当前主流是 **Python 和 TypeScript / JavaScript** 两个生态——**你已经会其中一个就够开始了**：

- 偏后端、偏数据、偏算法对接的项目，Python 更常见。
- 偏前端、偏 Web 产品形态的项目，TypeScript 更主流。
- 我自己工作中复杂 Agent 项目通常是 **Python 后端 + TS 前端**——两个都懂一些最舒服，但不是必须。

**必须会的：** Python 或 TypeScript 任一一种，能熟练写函数、处理 JSON、调一个 HTTP API。

**不需要会的（先放下）：** 深度学习数学、PyTorch / TensorFlow / 模型训练 / 微调、Transformer 内部细节、LangChain / LangGraph 等 Agent 框架的 API（**第一个 Agent 不应该从这些开始**）。

**关于 Git 和命令行：** 这些是工程师的"基础卫生"，但它们的学习曲线已经被 AI 工具大幅压平——遇到具体操作不会，问 Cursor / Claude Code，它会一步一步带你做。**不要因为不熟悉 Git 就觉得"还没准备好开始学 Agent"，本末倒置。**

**零基础补课清单（起点 A 专用，预计 1–3 周）：**

- **Python**：官方 Tutorial 前六章，或哈佛 CS50P（免费）——目标：会写函数、用字典、处理异常。
- **JSON + HTTP**：MDN 上《JSON》和《HTTP 概述》两篇——目标：知道请求 / 响应 / 状态码 / 请求体是什么。
- **Git**：GitHub Skills 的 "Introduction to GitHub"（交互式）——目标：会 clone / commit / push。
- **命令行**：会 cd / ls / 运行脚本 / 设置环境变量（存 API key 要用）。

补到什么程度算够？一个可自测的标准：**能把一个 JSON 文件读进来、改一个字段、写回去，并把这段代码 push 到自己的 GitHub 仓库**——就可以进阶段一了。

### 4.2 三阶段学习路线

先重申 v1 的一个判断，这一年它只被加强了：我**不建议**新人从零手写 ReAct Loop——那种"手写"在三年前是必经之路，但今天官方 SDK 的抽象已经足够轻，文档本身就是最好的教学。

**阶段一：跑通 SDK（3–5 天）**

如果你完全零基础，第一周的唯一目标是**手动跑通一次 LLM API 调用**。**这一周结束你拥有的最大资产不是代码，是"LLM 不是黑盒、它就是一个能调的 HTTP 服务"的真实体感**。

然后强烈建议直接读 **Vercel AI SDK 的官方文档**——它的文档本身就是一份非常优秀的"Agent 教学"，从"调一次模型"到"流式输出""调用工具""多步循环""结构化输出"是层层递进的。为什么不推荐 OpenAI Cookbook、Anthropic Cookbook 起步？因为它们各自只覆盖自家模型，**Vercel AI SDK 不限模型，迁移成本最低**。

Python 路线：用 **OpenAI SDK + base_url 模式**接 OpenAI 兼容的国内模型，或者用 **Anthropic SDK** 接 Claude。Pydantic AI 是对标 Vercel AI SDK 的轻量级 Python SDK。

> **前置**：补课清单达标（会写函数、处理 JSON、调 HTTP API）。
>
> **任务**：注册 DeepSeek API（V4-Flash 每百万 token $0.14/$0.28，充 5 美元都花不完）→ 用 10 行以内的代码把"你好"发给模型 → 加上多轮对话 → 跟着 Vercel AI SDK 文档跑通流式输出和第一次工具调用。
>
> **最小交付物**：一个能多轮对话的命令行脚本，放上 GitHub，README 写清怎么跑。
>
> **验收标准**：你能不查资料回答——为什么每一轮都要把完整的 messages 数组发回去？"模型记得上文"的本质是什么？
>
> **卡住了怎么办**：把报错原文直接丢给 AI 问（这是具体问题，AI 非常擅长）；检查 API key 和 base_url；对照官方 quickstart 逐行找差异。

**阶段二：理解机制（1–2 周）**

给阶段一的 Agent 加几个真实工具——网页搜索（Tavily / SerpAPI）、文件读写、第三方 API。重点观察**多工具下模型的"选择行为"**：什么时候选错？什么时候陷入死循环？把所有失败案例记下来——这些笔记是最好的简历素材。

然后做两个对比实验：试一次 **LLM Wiki / agentic search 模式**（一份资料整理成 Markdown 直接塞进 system prompt，或让模型按需 grep），再试一次 **RAG 模式**（同样的资料用 pgvector / ChromaDB 切片向量化）。**亲手做这种对比，比读 10 篇博客都管用。**

> **前置**：阶段一的交付物能跑。
>
> **任务**：加 2–3 个真实工具 → 刻意收集 10 个失败案例（选错工具、死循环、参数错）→ 完成上面两个对比实验。
>
> **最小交付物**：一篇《我的 Agent 是怎么失败的》笔记（发到博客或小红书——这是你的第一份公开输出），附对比实验的效果和体感结论。
>
> **验收标准**：能画出一次工具调用的完整生命周期（schema 定义 → 模型输出调用意图 → 程序执行 → 结果回填上下文）；能说出你的 Agent 在什么条件下最容易选错工具。
>
> **卡住了怎么办**：失败不是障碍——失败案例本身就是本阶段的产出；真卡死的问题，做一个最小复现，带着它去问。

**阶段三：真实项目（1–2 个月）**

挑一个**你自己每天会用的真实场景**——不要做"通用问答助手"这种烂大街的项目。

判断项目"够不够格"的标准——参考我那篇《模拟面试》博客里 W 的灵魂提问：**"如果是我，用豆包 / ChatGPT 直接就能解决，为什么一定要做这个？"答不上来，请换一个场景。**

几个有"简历价值"的入门项目方向：AI 选题助手（抓取小红书/Twitter 的热门内容做选题建议）、个人 Newsletter 助手（每周自动摘要订阅）、简易 Chat-to-SQL、个人邮件分类 Agent、轻量代码 Review 助手（接 GitHub Webhook）。

> **前置**：阶段二完成，且场景能回答"为什么不用现成 ChatGPT 解决"。
>
> **任务**：两周内做出最小可用版本（MVP）→ 给它写 5–10 条 eval（输入 + 期望行为，人工打分就行，先别追求自动化）→ 让至少一个真实用户（朋友也算）用起来 → 根据反馈迭代。
>
> **最小交付物**：一个别人可以真的用起来的项目，加一篇复盘博客（"做了什么 / 踩了什么坑 / 怎么解决"）。
>
> **验收标准**：能扛住 W 的灵魂拷问；复盘里至少有一个带数字的改进（哪怕是"重试上限从无限改成 3 次，死循环成本降了九成"这种小数字）。
>
> **卡住了怎么办**：把范围砍一半，再砍一半——新人项目死于贪大，不死于简陋。

**整条路线的成本预算**：用 DeepSeek 走完三个阶段，API 花费大概率不超过 **100 元人民币**；想在阶段三体验 Claude / GPT 旗舰，再备 100–200 元。**穷，不是入门 Agent 的障碍。**

**一条并行路径（强烈建议）**：从阶段一开始，**日常同时深用一个成熟 harness（Claude Code 或 Codex）**。不只是让它帮你写代码——观察它：上下文怎么组织？权限怎么向你申请？什么时候压缩历史？工具怎么描述？这是你能免费看到的、最工业级的 harness 活教材，和你自己手搓的小 Agent 互为参照。

每个项目完成后**写一篇博客复盘**——"我做了什么""遇到了什么坑""怎么解决的"。**这篇博客本身就是简历最强的素材。**

### 4.3 求职准备：项目要"讲到极致"

如果你只能把准备时间花在一件事上，**就是把你的项目讲到极致**——说清楚四件事：

- **做了什么**（What）：项目背景、你的角色、整体架构
- **为什么这么做**（Why）：每一个关键决策的依据
- **踩过什么坑**（How it failed）：失败案例 + 解决方案
- **学到了什么**（What you learned）：重做你会怎么改

**反例**："我用 LangChain 做了一个 RAG 客服系统。"——这种描述等于没说，是面试杀手。

**正例**：

> "我做了一个客服 RAG 系统。最初用简单向量检索，召回率只有 60%——分析发现客户问题口语化，和文档原文的措辞差距大。我们引入了 Query Rewriting：先用轻量模型把用户问题改写成多个候选 Query，再分别检索后合并去重。这一改召回率提升到 85%，但 Token 成本增加 30%。为了平衡成本，我们后来加了缓存——同一类问题的改写结果可以复用。最终在召回率 80%+ 的前提下，成本只增加 5%。"

这段话有：**指标（60% → 85%）+ 决策（Query Rewriting）+ 权衡（成本 vs 召回）+ 后续优化（缓存）**。这就是"讲到极致"。

**简历层面的三个要点**：不要堆技术名词（"熟练掌握 LangChain、LangGraph、Vercel AI SDK、CrewAI……"这种简历看了让人发笑——真正强的候选人简历里技术词反而少）；每段项目按"问题 — 方案 — 结果"结构；数字化成果（哪怕估算也比没数字强）。

**面试层面**——Agent 岗考的不是八股文，是"你经历过什么"。具体四个维度：

1. **基础认知**：LLM / Agent / Chatbot 的本质区别、Function Calling 怎么工作、MCP 是什么……
2. **系统设计**：上下文工程方案、记忆分层、工具调用可靠性、多 Agent 协作……
3. **工程取舍**：模型选型依据、成本和效果平衡、框架选型判断、失败重试策略……
4. **行业认知**：Claude Code / Codex / OpenClaw 的设计思路、harness 和 loop 这些新词在说什么、最近读了什么、关注什么开源项目……

第一层靠经验、第二层靠理解、第三层靠判断、第四层靠 taste 和阅读量。**越往后越能拉开候选人差距。** 我那篇《模拟面试》博客里把每一类都举了具体例子，需要的话可以去看。

### 4.4 五个不要走的弯路

**弯路一：一上来就啃 LangChain 源码。** 设计复杂，源码对新人极不友好。等你用 SDK 跑过几个项目之后再去看，体感会完全不一样。

**弯路二：过早卷模型微调。** 99% 的应用场景不需要微调，Prompt 工程 + agentic search / RAG 已经能解决大部分问题。

**弯路三：追新框架不沉淀基本功。** 每隔两周就有新框架。一旦你形成"追新"习惯，就永远在学新东西、永远没有自己的项目。

**弯路四：看完教程就觉得自己会了。** Agent 这个领域所有"看起来很简单"的概念，真正动手都会发现一堆细节。**只看不写等于零。**

**弯路五：只做不输出。** 做了项目不写文档、不写博客、不发开源，等于白做。**输出是倒逼输入最有效的方式**，也是你后续找工作时最强的差异化资产。

### 4.5 推荐学习资源（精选）

**Harness / Loop 专题（v2 新增，读完你对这个领域的理解会上一个台阶）**：

## 第五章｜真正重要的那几件事

### 5.1 上下文工程

**类比**：你给同事交接工作，给的是 100 页项目档案，还是一份 1 页精要 brief？

LLM 的注意力是有限的——上下文越长、信息密度越低，它越容易"看走神"，同时 Token 成本越高、响应越慢。**Context Engineering 解决的就是"在有限空间里，把最该被看到的信息以最有效的方式呈现给模型"。**

在实际项目里，一个商业级 Agent 一次对话可能要处理几十轮交互、调用几十次工具。如果不做上下文管理，10 分钟内就会把上下文撑爆。常见技巧——

- **结构化 Prompt**：用 XML 标签、JSON 块、明确分隔符代替自然语言流水账。
- **关键信息前置 / 后置**：模型对开头和结尾的注意力更高（"Lost in the Middle"现象）。重要约束放 System Prompt 顶部或 User 消息末尾。
- **历史摘要替代历史明文**：长对话里把早期对话压缩成摘要（2026 年起主流 API 已提供服务端自动压缩，比如 Anthropic 的 compaction）。
- **Prefix Cache 友好的上下文设计**：不变的内容放前面、变化的放后面，能极大降低成本。

### 5.2 记忆系统

**类比**：人怎么记事？短期记忆（刚发生的事）、长期记忆（多年前的重要经历）、检索唤起（看到老照片突然想起一段往事）。Agent 的记忆架构基本是仿这个的。

**三层架构**：

- **Working Memory**：当前任务正在用的上下文
- **Short-term Memory**：当前会话的历史
- **Long-term Memory**：跨会话的持久信息

长期记忆的关键决策点有三个——

**写入策略**：什么样的信息值得写入？"我今天想吃辣"这种临时偏好不该记；"我对花生过敏"这种长期事实必须记。这种分类通常由专门的"Memory Agent"来判断。

**读取策略**：什么时候检索、怎么检索？每次对话都检索一次还是特定意图下检索？用向量相似度、关键词、图检索？

**遗忘策略**：长期记忆不是越多越好。过时、低价值、矛盾的记忆应该被清理或衰减。

顺带一个 2026 年的实践观察（2.4 也提过）：**实践里赢的长期记忆方案，大多是"文件式 + agent 自己管理"**——Anthropic 的 memory tool、CLAUDE.md、LLM Wiki 全是这个形态。动手做记忆系统时，别默认第一步就上向量库。

### 5.3 工具调用

**类比**：让一个聪明但没有手的人帮你完成任务——你得告诉他附近有哪些工具、每个能做什么、怎么用。

工程上几个常见难点：

- **Tool Schema 设计**：参数名、描述写得越清楚，模型用错概率越低。
- **工具数量取舍**：太少不够用，太多模型选不对。**"10 个左右"是个常见的经验起点，不是定律**——真实上限取决于模型能力和工具描述的区分度；超过之后就需要"工具路由"（正是 Skills 的按需暴露解决的问题）。
- **失败重试与幂等**：失败重试是必要的，但要有上限——**"3 次"是常用经验值**，合理的重试策略取决于错误类型和幂等性：超时可以重试，"余额不足"重试一万次也没用。
- **Prompt 约束和执行层权限，是两道不同的门，都要装**（v2 重写了这一条，v1 的说法有安全隐患）：在 Prompt 里把工具使用边界讲清楚，能显著降低误调用率、省掉很多兜底成本——但它**不是安全边界**。任何有副作用的工具（下单、发消息、写数据库、删文件）必须在**工具执行层**做硬控制：白名单、参数校验、金额/范围上限、执行前确认、幂等键、审计日志。一句话：**Prompt 管"少犯错"，程序管"不能错"——模型的自觉永远不是安全机制。**

### 5.4 可靠性

**类比**：写 Demo 像在自家厨房做菜，写 Production 像开餐厅——你要应对的不只是"做得好不好"，还有"高峰期会不会爆""偶尔来个挑剔顾客会不会让流程崩"。

传统应用（至少在业务逻辑层）近似"确定性系统"——同样输入得到同样输出。Agent 是"概率性系统"——同样输入可能得到不同输出，甚至完全失败。**这意味着"测过一次就 OK"的开发模式在 Agent 上完全行不通。**

常见可靠性问题：幻觉、指令偏离、格式不稳定、死循环、工具失败的级联崩溃。

工程上的核心思路：

- **事前约束**：用 Prompt 把"应该怎么做"讲清楚，比事后兜底成本低（注意：这说的是减少错误，不能替代 5.3 里的执行层权限控制）
- **结构化输出 + Schema 校验**：用 Pydantic、Zod 校验模型输出
- **状态机 + Checkpoint**：把 Agent 流程显式化为状态机
- **降级策略**：工具失败时有 fallback 路径

### 5.5 成本控制

**类比**：开车——油价、路程、车型都会影响油钱。Agent 也一样：模型、上下文长度、调用次数共同决定一次任务的成本。

Agent 的成本远比传统应用高。一次复杂任务可能要几十次 LLM 调用、累计几万到几十万 Token——单次任务可能花几块到几十块人民币。**如果你的产品是 toC 免费的，成本控制不好就是赔本赚吆喝。**

几个高 ROI 的优化手段：

- **Prefix Cache 友好设计**：主流厂商都对"前缀命中"给大幅缓存折扣（Anthropic 的缓存读取约为原价一折；具体以各家 pricing 页为准）。把不变的内容放前面。
- **模型分级使用**：简单任务用便宜模型，复杂任务才用旗舰。
- **减少 Agent Step 数**：能在一步讲清楚的事就不要拆成多步。
- **上下文剪枝**：把不相关的工具结果、过时对话历史从上下文剔除。

### 5.6 评估（Eval）

**类比**：传统软件能用单元测试——输入 1+1，期望输出 2，错了就是 Bug。Agent 在很多任务上没有唯一"标准答案"——你怎么知道它做得"好"？

不过先修正一个 v1 说得太满的表述：Agent"没有标准答案"，**不等于"没有可判定的约束"**——订错票、泄露数据、调用了禁用工具，都是明确的错。所以 Eval 体系要同时覆盖两层：**硬约束（必须全过的测试）+ 质量光谱（打分比较的部分）**。有了这套机制，你才能回答"我新版本的 Agent 比上一版好还是坏？"——没有它，你优化半天根本不知道方向对不对。**Eval 体系是 Agent 工程从"作坊"到"工业"的标志。**

主流评估方法：

- **离线评估**：准备一批测试用例，让 Agent 跑，人工或 LLM-as-Judge 打分
- **在线评估**：在生产环境里收集真实用户反馈（点赞点踩、停留时长、是否继续追问）
- **LLM as a Judge**：用更强的模型当裁判——注意它本身的偏差（倾向打高分、偏好长答案等）
- **对照实验**：A/B Test，把新版和旧版分流给不同用户

**2026 年的行业注脚：Eval 已经从"内功"变成了"风口"。** OpenAI 的 CPO 说"写 evals 是 AI 时代产品经理最重要的技能"；OpenAI 在 3 月直接收购了评测公司 Promptfoo；评测平台 Braintrust 拿了 8000 万美元 B 轮；腾讯研究院《超级个体时代》报告列的十个 AI 原生工作关键词里，"评估"赫然在列。**对求职者来说，"会做 eval"是当下性价比最高的差异化技能——给你实习或自己的项目做一套测评，是我能给的最实在的弯道超车建议**（展开见[《从实习面试到 Agent 开发》](https://www.joyehuang.me/blog/20260615---frominternshipinterviewstoagentengineering/post)的"去做测评"一节）。

### 5.7 这六件事是 Agent 工程师真正的"内功"

总结一下——

1. **上下文工程**：有限空间里最大化信息密度
2. **记忆系统**：让 Agent 像人一样分层记事
3. **工具调用**：让 Agent 能"动手"——并且不能乱来
4. **可靠性**：从确定性思维切换到概率性思维
5. **成本控制**：钱是真的会烧的
6. **评估**：没有 Eval 的 Agent 优化都是玄学

如果你能在简历或面试中讲清楚这六件事，你已经领先大多数应聘者了。
