## Chapter 1 | Getting to Know AI Agents

> Before you can "develop Agents," you first have to be able to "read Agents." This chapter helps you explain in three sentences what an Agent is and how it differs from the things you already know (ChatGPT, APIs, chatbots).

### 1.1 An imprecise but easy-to-grasp definition

> **If the large language model (LLM) is a "brain," then an Agent is that brain with eyes, hands, memory, and a body that lets it work in a loop on its own.**

A slightly more rigorous statement: **an Agent is a system with an LLM as its decision-making core that can perceive its environment, call tools, maintain memory, and complete multi-step tasks through an autonomous loop.** Four keywords: **LLM, perception, tools, loop**.

### 1.2 What an Agent is not: three common misconceptions

**Misconception 1: Agent ≠ chatbot.** The core of a chatbot is "conversation"; the core of an Agent is "completing a task" — conversation is just one of the ways it receives a task.

**Misconception 2: Agent ≠ a single LLM API call.** A single-turn call is "input → output"; an Agent has to have at least one of these to count as an Agent: **a loop, tools, state**.

**Misconception 3: Agent ≠ a form-filling app wrapped around an LLM.** Stuffing an LLM into a product where the user fills out a form is no different in essence from "a search box with GPT-4 bolted on" — because the model participates in none of the "what to do next" decisions.

But right here I need to correct something v1 stated too strongly (a common flaw in intro material): **"the model decides every step by itself" is not an entrance exam for being an Agent — autonomy is a spectrum.** The most common shape in real production systems is precisely "a deterministic workflow controls the backbone, and the LLM only decides at local nodes like classification, planning, and tool selection" — with permissions, retries, approvals, and state transitions controlled by the program. Deliberately **reducing** the model's degrees of freedom is often the correct reliability engineering, not a "fake Agent." Anthropic's own guidance treats workflows and agents as one spectrum you pick from as needed. So the more accurate statement is: the form-filling app's problem isn't "not autonomous enough," it's that the model participates in no decisions at all; **how much autonomy a good system grants the model is an engineering trade-off, not an identity test.**

One thing to add: back in the early ChatGPT of 2023, you gave it an input and it gave you an output, and that was it. By 2025–2026, the mainstream web versions of ChatGPT, Claude, and Gemini have built quite a few Agent-ified capabilities into the product — web search, code execution, file read/write, MCP tool calls. **But this is "vendors packaging Agent capabilities at the product layer for users to use," not "the LLM itself becoming an Agent."** The underlying model is still that brain; it can act because there's a whole stack of Agent engineering wrapped around it — and that whole stack is exactly what you're going to learn.

### 1.3 The Agent four-piece set — and the Harness that assembles it

Understand these four pieces and you've understood 80% of the engineering substance of Agents.

**Brain (LLM)**: thinking and decision-making. It takes in the current state and outputs the next action (answer the user / call a tool).

**Memory**: build the intuition in two layers first — **short-term memory** is the context of the current task (what the user just said, what tool was just called); **long-term memory** is persistent information across sessions (user preferences, key facts). In engineering practice it's actually further divided into three layers — Working / Short-term / Long-term — which Chapter 5 expands on.

**Hands (Tools / Function Calling)**: the Agent's interface for interacting with the outside world — search engines, code execution, email APIs, database queries are all tools. All the major LLM vendors provide native Function Calling.

**Loop (Agent Loop)**: `perceive → think → act → perceive → ...` The most classic implementation is called **ReAct** (Reason + Act). The termination condition is usually: the model decides it's done / it hits a max step count / the user interrupts.

And the layer of engineering that **assembles** these four pieces — all the scaffolding that decides what the model sees at each step, what it's allowed to touch, and what happens when it errs — got a formal name in 2026: the **Harness**. Vivek Trivedy, the LangChain engineer responsible for its open-source agents, has a line worth memorizing: **"Agent = model + harness. Everything that isn't the model is the harness."**

That's not an empty slogan. In February 2026 LangChain [ran an experiment](https://www.langchain.com/blog/improving-deep-agents-with-harness-engineering): without touching the model at all (it was the closed-source GPT-5.2-Codex), they rebuilt only the harness — system prompts, tools, loop-detection middleware — and lifted their agent's Terminal-Bench 2.0 score from 52.8% to 66.5% (+13.7 points, which put it in the leaderboard's top five at the time). **A lot of problems that look like "the model isn't good enough" are actually "the harness isn't built well."** Everything you'll learn in the rest of this document — context, memory, tools, reliability — is at bottom learning how to build a good harness.

### 1.4 A minimal Agent workflow: using "order me lunch" as an example

You say to an Agent: "Order me a lunch, under 30 yuan, ideally Cantonese." Internally it runs roughly like this:

**Round 1**: thinks "first look up nearby Cantonese restaurants" → calls `search_restaurants(cuisine="Cantonese")` → gets 8 restaurants.

**Round 2**: thinks "filter by budget" → calls `filter_by_price(max_price=30)` → 3 left.

**Round 3**: thinks "I have enough info, give it to the user" → answers directly: "I found 3 — A, B, C. Which would you like?"

Note four key points:

1. **Every step is decided by the model itself** — it judges on its own whether to call a tool, which one, and when to stop. This is the most central difference between an Agent and a "hardcoded workflow" (though, per 1.2, in real systems this freedom is deliberately designed).
2. **The tool execution is not done by the model** — the model only outputs "I want to call this tool"; the actual execution is your program's job.
3. **Memory runs through the whole flow** — by the time it's thinking in round 3, the model still remembers the original budget and taste requirements.
4. **Actions with real side effects must pass a program-layer gate** — when it actually gets to "place the order and pay," a responsible Agent will confirm with you first; and boundaries like "may it order at all, what's the amount cap" are enforced by the program at the tool-execution layer, never left to the model's self-restraint. This matters a great deal — Chapter 5 devotes space to it.

### 1.5 This chapter in one sentence

> **Agent = an LLM in a loop, repeatedly "thinking a bit, doing a bit," until the task is done — and all the engineering around that loop is called the harness.**

## Chapter 2 | A Map of the Agent Ecosystem

One thing up front: the "five-layer map" below is **a memory aid, not a rigorous architecture spec** — MCP and Skills don't actually sit at the same level of abstraction, RAG and Memory overlap, and the boundaries in real systems are far blurrier than the diagram. ### 2.1 The model layer: which brain to choose

First, a lesson learned: v1 filled this section with specific model names, and **every one of them was stale within eight weeks** — that's just how the model layer works. So this version changes the format: first the landscape and selection logic (which don't expire), then the specific models in a snapshot table stamped with a date. **Before actually choosing, check each vendor's official models page and the public leaderboards — not any static article, including this one.**

**The landscape (slow-changing)**: three overseas closed-source flagships (Anthropic's Claude, OpenAI's GPT, Google's Gemini) + five Chinese labs (DeepSeek, Qwen, Kimi, GLM, MiniMax). The overseas camp competes on flagship capability and agent-engineering maturity; the Chinese camp competes on cost-performance and open weights. So v1's "for a Coding Agent, Claude is the default choice" gets demoted to: Claude remains the reputational default for agent-engineering ecosystem, but **don't outsource the decision to any static conclusion — check the current leaderboard.**

**Four pieces of honest advice on model selection (these don't expire)**:

1. **Bigger isn't better — use models tiered by task.** Use a cheap small model for simple intent recognition and routing; only use the flagship for complex code generation and reasoning. This is the key technique for controlling cost.
2. Most Chinese models provide an **OpenAI-compatible API endpoint** — the same code can switch models just by changing `base_url`.
3. **Don't bet everything on a single model** — a mature project will plug into 2–3 vendors at once and route by task.
4. **The weight of model selection is falling.** Remember the experiment in 1.3: model untouched, harness rebuilt, +13.7 points. The gap between same-tier models is often smaller than the gap between harness qualities — the time you'd spend agonizing over model names pays better spent on tool design and context engineering.

### 2.2 The framework / harness layer: what scaffolding to build with

The mainstream choices fall into two categories: **official SDKs / harnesses** and **third-party frameworks**.

**The official camp:**

- **OpenAI SDK**: the oldest and most stable. **But it can only call OpenAI's own models** — though, because the OpenAI-compatible protocol is the de facto standard, Chinese models like DeepSeek / Qwen / Kimi / GLM can all be called with it by changing `base_url`.
- **Anthropic SDK**: Claude's official SDK, and the smoothest way to wire up native capabilities like Tool Use and Computer Use.
- **Claude Agent SDK** (renamed from the Claude Code SDK in September 2025): note this is not the same thing as the one above — it isn't a "call the model" SDK but a complete harness that **"gives the agent a computer"**: a built-in filesystem, terminal, permission system, and the gather-context → act → verify loop. If you want to build on a mature harness directly, this is one of the smoothest entry points today.
- **Google Gen AI SDK**: Gemini's official SDK, with the most direct multimodal integration.

**Third-party frameworks (in order of recommendation):**

**Vercel AI SDK** (my personal top pick for getting started — unchanged from v1, and if anything strengthened):

- **Model-agnostic** — `import { anthropic } from '@ai-sdk/anthropic'` to use Claude, and switching to OpenAI / Google / DeepSeek only changes a single import while the rest of your code stays completely untouched.
- **Light abstractions** — it gives you primitive-level capabilities like streaming, tool calling, and structured output, and won't force you to organize your code around its abstractions (Chain / Graph / Node) the way LangChain / LangGraph do. How you orchestrate the Agent Loop is entirely up to you.
- In AI SDK 7 (June 2026), agents are first-class citizens: ToolLoopAgent (the tool loop), WorkflowAgent (durable long-running tasks), and notably **HarnessAgent — which wraps ready-made harnesses like Claude Code, Codex, Deep Agents, and OpenCode behind one interface, so you can swap harnesses the way you swap models**. (Note v7 requires Node 22 + ESM.)
- The TypeScript / Next.js ecosystem is especially smooth.

**LangChain / LangGraph**: the most established, the biggest ecosystem, the most extensive docs — but also the most frequently criticized for "abstractions that are too heavy," and that criticism still holds in 2026. Notably, LangChain's own center of gravity has visibly shifted to **deepagents**, whose official tagline is "the batteries-included agent harness." Unless your project specifically needs LangChain's off-the-shelf components, **I don't recommend it as a first stop.**

**Pydantic AI**: the lightweight Python-side counterpart to the Vercel AI SDK; its V2 (June 2026) is explicitly "harness-first" by design. **CrewAI / AutoGen / Mastra**: aimed respectively at multi-Agent collaboration, enterprise-grade orchestration, and TypeScript full-stack — pick one once you have a concrete scenario.

**Advice for beginners** — there are now two validated paths, and they don't conflict:

- **Path one (from v1)**: write your first Agent with the **Vercel AI SDK or a vendor's official SDK** — the abstractions are light enough not to block your view of "what the Agent Loop is actually doing."
- **Path two (repeatedly validated through 2026)**: **from day one, use a mature harness (Claude Code or Codex) daily and treat it as a living textbook** — watch how it organizes context, how it designs permissions, when it compacts history. Philipp Schmid and Addy Osmani both argued in dedicated 2026 essays that newcomers should start from a harness. Osmani's line is worth remembering: **"a decent model with a great harness beats a great model with a bad harness."**

**Don't learn a framework for the sake of "learning a framework"** — the framework itself isn't a résumé asset; "what valuable thing I built with some framework" is.

### 2.3 The protocol layer: MCP and Skills

**MCP (Model Context Protocol)** is an open protocol Anthropic launched in late 2024 that gives "LLM applications" and "tools / data sources" a **standardized way to talk to each other**.

An analogy — in the past, every Agent that wanted to plug in a new tool had to write its own adapter code; MCP is like the "USB standard" of the Agent world. And here's an important v2 update: **in December 2025, Anthropic donated MCP to the newly formed Agentic AI Foundation under the Linux Foundation (co-founded with OpenAI and Block)** — it went from "one company's protocol" to an industry standard under neutral governance, and the official registry now lists roughly ten thousand MCP servers. One more date worth noting: MCP's largest spec revision ever (a stateless redesign) is scheduled to land on **2026-07-28** — it may have just been finalized as you read this.

The beginner advice is unchanged: first **use MCP Servers other people have already written** and plug tools like Notion / GitHub / Slack directly into your Agent.

**Skills is an equally important concept to understand.** It's a **product form** Anthropic launched in October 2025 — making a "packaged, reusable Agent capability module" a first-class citizen. v1 said "the industry is starting to follow"; that can now be written in the past tense: **Skills became an open standard in December 2025 ([agentskills.io](https://agentskills.io)), and by July 2026 it has 40+ adopting clients** — including OpenAI Codex, Gemini CLI, GitHub Copilot, VS Code, and Cursor.

To understand Skills, first look at an engineering reality: a genuinely usable Agent often isn't just "a model + a few tools." It also needs — a set of **dedicated tools** (generating a PPT needs a pptx library), a piece of **carefully tuned prompt guidance** (when to use what, what the pitfalls are), some **examples and reference materials** (so the model knows "what good output looks like"), and sometimes **specific code snippets** as well.

If these are scattered across the prompt, tool descriptions, and code comments, two things happen: the model doesn't know when to use what; and these capabilities can't be reused, distributed, or versioned.

**Skills turns this "capability package" into a product form that can exist independently, be loaded, and be shared.** A Skill is usually a folder containing `SKILL.md` (describing what this Skill does and when it triggers) + related scripts, tools, and reference materials. The Agent automatically loads the corresponding Skill when it needs it.

**A few key intuitions:**

- Skills are a **"user manual" for the Agent to use**, not documentation for humans to read — the language is written for the model.
- Skills solve the problem of "infinite tools confusing the model." If an Agent plugs in 50 tools, the model is extremely likely to pick the wrong one; but if you group them into 10 Skills, the model only sees the corresponding tools in scenarios that match that Skill — this is **on-demand exposure**.
- **Skills and MCP are complementary**: MCP solves "how a tool gets called" (interface standardization); Skills solves "how tools get organized and triggered" (capability packaging). A single Skill can internally call multiple MCP Servers.

**The relationship among the three:**

- **Function Calling / Tool Use** actually refer to the same thing — OpenAI calls it Function Calling, Anthropic calls it Tool Use, and at bottom both are the underlying capability of "can the LLM output the instruction 'I want to call this tool.'"
- **Skill** is another layer of abstraction — packaging a set of tools + prompt guidance + reference materials into a reusable, loadable capability module.

Two more names to file away so JDs don't surprise you: **AGENTS.md** (the repo-level agent-guidance file convention, used in 60,000+ open-source repos) and **A2A** (an agent-to-agent communication protocol backed by 150+ organizations).

### 2.4 The data layer: RAG, Agentic Search, Memory

This layer is changing fastest in 2026 — the traditional RAG paradigm is being partially replaced by several new forms.

**RAG (Retrieval-Augmented Generation)** was the most mainstream way of "giving an LLM its own knowledge" in 2023–2024. The core component is a **vector database** (Milvus, ChromaDB, Pinecone, pgvector, sqlite-vec).

Go back to the era when RAG emerged (early 2023): mainstream LLMs had a context window of only 4K–32K tokens, long documents couldn't fit, and the only option was "chunk + vector retrieval + stitch back into the prompt." Today mainstream flagships routinely support 1M tokens — for many scenarios the complex "retrieve first, then generate" pipeline isn't necessary at all, and stuffing the document straight into the context works better.

By mid-2026, the "is RAG dead" debate has mostly converged on one sentence: **naive "chunk + vector retrieval" is no longer the default architecture, but retrieval itself lives on happily as one of the Agent's tool actions.** The new pattern is called **Agentic Search (agent-as-retriever)**: build no index; let the agent explore on demand with grep / keyword search. Two landmark pieces of evidence: Claude Code still builds no vector index over your codebase — it searches live with filesystem tools; and an AWS paper at AAAI 2026 is literally titled ["Keyword search is all you need"](https://arxiv.org/abs/2602.23368) — pure keyword-based agentic retrieval reaches 90%+ of RAG's performance. But you should see the counter-evidence too: [Cursor's official experiments](https://cursor.com/blog/semsearch) show that **stacking** semantic search on top of grep still improves answer accuracy by an average of 12.5%. So the takeaway I'll teach you is: **grep / agentic search first; add embeddings when scale and recall demand it — the two stack, they don't replace each other.**

**LLM Wiki** is another pattern worth knowing in this direction: in early **April 2026** (v1 mistakenly said May — corrected), Karpathy published [the idea](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) — for an individual's or small team's medium-scale knowledge base, drop vector retrieval entirely; organize everything as a Markdown "Wikipedia" that the LLM incrementally maintains and stuffs into context on demand. Upsides: no chunking errors, no recall problems, trivially easy debugging, and Prefix-Cache-friendly. That said, the first long-term field tests two months later reached a sober conclusion: it works for narrow cases, but maintenance time roughly equals time saved. **Don't romanticize it — treat it as a pattern worth trying with your own hands** (Chapter 4 has you compare it yourself).

**But traditional RAG isn't dead.** It's still the top choice in these scenarios:

- **Internal enterprise scenarios that emphasize data privacy** — documents can't leave the corporate network.
- **Ultra-large-scale knowledge bases** (millions of documents and up) — the context can't hold them and grep can't sweep them.
- **Multi-user, multi-tenant scenarios** — each user's data has to be isolated.

In short: **for a personal project, try agentic search / LLM Wiki first; for a toB enterprise project, RAG is still the default.**

**Memory (the memory system)** overlaps with RAG but is different — Memory emphasizes "personalized memory of the current user / session," usually in three layers: Working / Short-term / Long-term (Chapter 5 covers it). In 2026 it graduated from concept to product layer: Anthropic's memory tool is GA (note that it's **file-based** — the agent reads and writes Markdown under a /memories directory, not an opaque vector store); OpenAI rebuilt ChatGPT's memory in June ("Dreaming" — memories update themselves as time passes); and on the open-source side Mem0 / Zep / Letta compete on memory benchmarks. **One pattern worth noticing: the memory designs that win in practice are mostly "file-based + agent-managed"** — the memory tool, CLAUDE.md, Karpathy's wiki — all the same shape.

### 2.5 The application layer: proven scenarios, plus three mid-2026 developments

After more than a year of observing the industry, the directions that get mentioned over and over and have proven out commercially are mainly these three:

**AI search**: from Perplexity to Metaso AI to Felo, all of these are essentially Agent-ified search — no longer "keyword matching + ranking," but "understanding intent + retrieval + synthesis + generation." This direction has proven it can charge money: Perplexity's annualized revenue hit roughly $450M by March 2026.

**Chat-to-BI**: letting business people query data, generate charts, and do attribution analysis in natural language. It's begun charging money in China too — Alibaba Cloud's DataWorks ChatBI switched to formal commercial billing in April 2026.

**Vibe Coding**: from Copilot to Cursor to Claude Code — the fastest-moving direction of the past year, and the one that most directly disrupted traditional software development. Two numbers for a sense of scale: Claude Code passed a $2.5B annualized revenue run rate in February; OpenAI's Codex passed 5 million weekly active users in June, **roughly 20% of them non-developers**.

Beyond those three, there are three mid-2026 developments worth knowing when you pick a direction:

**One: Agents broke out of the developer niche.** Anthropic's own data for Claude Cowork (the desktop agent for non-technical workers) shows 90%+ of sessions are non-coding work; OpenClaw (the open-source gateway that connects a personal agent to WhatsApp / Telegram / WeChat) rocketed to 380K GitHub stars, one of the fastest star curves in GitHub history. The "agents = programmer toys" era is over.

**Two: general vs. vertical — the scale is tipping toward vertical.** General agents are the battleground of model vendors and platforms — Meta's acquisition of general-agent star Manus was blocked by regulators in April and formally abandoned in June, and Manus itself is pivoting to vertical task packs; meanwhile verticals are making real money: legal agent Harvey at roughly $300M ARR, customer-service agent Sierra past $100M ARR. The jargon for this logic is the **context moat**: domain context + workflows + outcome delivery are the parts a model update can't compete away. For an individual developer, the opportunity is in verticals — I went deeper on this in [my second fan-group talk](https://www.joyehuang.me/talks/2026-ai-product-form.html) (deck in Chinese).

**Three: the product form is changing generations.** In June, Karpathy offered a three-stage arc: LLM products went from "a website you visit (ChatGPT) → an app on your machine (Claude Code / Cursor) → **a persistent entity living inside an organization**." The emblem of stage three is Claude Tag, which Anthropic launched in late June: a Claude that lives in a Slack channel, has memory, takes asynchronous tasks, and is shared by the whole team. What it means for you: **the thing an "Agent engineer" builds is shifting from "a chat window" to "a digital coworker."**

## Chapter 3 | How to Think About This Direction: Trends, Mindset, and Pitfalls

> This chapter isn't about technology. It's about three things: why Agents are the most worthwhile direction to get into right now, why you don't need to be anxious, and how to avoid the opportunities that look beautiful but are actually traps.

### 3.1 "Am I too late" is a false question

The question I get asked most in consulting is: "Joye, am I too late if I'm only starting to learn now?"

My standard answer is: **even the "veterans" on this track only have two years of experience.**

A quick sketch of the timeline:

- **November 2022** ChatGPT released
- **Early 2023** open-source projects like AutoGPT take off, and "Agent" starts being widely discussed
- **June 2023** OpenAI launches Function Calling, and Agent engineering enters a new phase
- **2024** Cursor enters its commercial explosion period
- **Late 2024** Anthropic launches MCP, and the Agent protocol layer starts taking shape
- **2025** Agent products like Manus, Claude Code, and Devin burst onto the scene all at once
- **October–December 2025** Anthropic launches Agent Skills; MCP is donated to the Linux Foundation; Skills becomes an open standard
- **First half of 2026** "Harness Engineering" becomes an industry term (Anthropic's engineering blog series, LangChain's +13.7-point experiment, Osmani's naming essay)
- **June 2026** "Loop Engineering" appears — one Steinberger post ignites it, Osmani systematizes it the same day; the same month, Claude Tag launches and agents enter their "organization-resident" form
- **July 2026** Lilian Weng publishes a survey of harness engineering — the field is getting its first "textbook"

In other words — **the so-called "senior Agent engineers" in the industry today have at most two to three years from entry to now.** Or, more pointedly: **nobody has "ten years of Loop Engineering experience" — the term appeared this June, and the most senior engineer read the same post you did, the same week you did.** Seniority expires; curiosity compounds. Compared with the directions in traditional development where people have ten or twenty years on you, this is a field you can genuinely **catch up in through speed of learning.**

### 3.2 Why now: three judgments

**Judgment 1: Agents are moving from the "Demo phase" to the "Production phase."**

In 2023 and the first half of 2024, a huge number of Agent projects in the industry stalled at the Demo stage. Starting in the second half of 2024, "industrial-grade" problems like reliability, observability, Eval systems, and cost control started being taken seriously — **this is the phase where an engineer can genuinely add value.**

**Judgment 2: a standardization window at the infrastructure layer.**

The standards for infrastructure layers like MCP, Skills, and AI Gateway are still taking shape quickly (MCP's biggest revision ever finalizes at the end of this very month). This means that if you enter now, **you have a chance to genuinely participate in building the infrastructure of a new industry** — and that kind of window is extremely rare in traditional development.

**Judgment 3: talent demand is growing systematically.**

In 2026 this judgment acquired hard numbers, on both sides of the ocean:

- **Y Combinator**: the W26 batch (Winter 2026) was roughly 60% AI companies with over 40% building agent infrastructure; by the **P26 batch (Spring 2026, Demo Day in June), companies building LLM agents already made up about 70% (137 of 196)**. When one of the world's most discerning incubators puts seventy percent of its chips on agents, talent demand in this track only goes up.
- **Chinese campus recruiting**: **Alibaba's class-of-2027 campus recruitment (launched March 2026) is 80%+ AI roles and, for the first time, created seven new AI job families including "AI Agent Optimization Engineer" and "Agent Infra Engineer"** — the first time "Agent engineer" has literally entered a big-tech campus job catalog; ByteDance Seed's 2027 foundation-model campus recruiting opened in April. Maimai's spring recruiting report (May 2026): AI job volume grew 8.7x year over year.
- **Overseas data**: Stanford's AI Index 2026 added "Agentic AI" as a tracked skill cluster for the first time, with job-posting mentions up 280%+ in a year; AI Engineer topped LinkedIn's fastest-growing jobs list for the second consecutive year.

### 3.3 A few reasons you don't need to be anxious

**First, the industry has no "absolute authority."** Traditional computer science has those "I've read his paper," "I've read his book" authority figures. This Agent direction is too new for that kind of figure to exist. OpenAI's and Anthropic's best practices are all written by engineers as they go — the gap between them and you is "accumulated practice," not "a gap in talent."

**Second, the information asymmetry is tiny.** OpenAI and Anthropic publish best practices on prompt engineering, Agent design, and the Skills system directly on their company blogs, **free for anyone to read**. That kind of transparency is unimaginable in traditional industries — most of what you want to learn has already been written down, and the only question is whether you're willing to spend the time reading it.

**Third, the "easy part" of the tooling barrier is dropping, but the "deep part" is rising.** This point needs to be told in two halves.

Looking at the easy side: three years ago, integrating an LLM required understanding GPU deployment; today you only need to know how to call an API. Vibe Coding has made "build a personal website" and "build a simple chatbot" nearly barrier-free.

But this is precisely why — **when building a "barrier-less Agent project" becomes easy, a barrier-less project is itself worthless.** You can find a thousand "AI health assistants" and "AI customer-service bots" on GitHub, because everyone can build one in a weekend with Vibe Coding. Put these projects on a résumé and the interviewer knows their worth at a glance.

The real entry barrier has been pushed to a **deeper place**: can you pick a **problem that genuinely exists and can't be solved with off-the-shelf ChatGPT** to build a project around? Can you make **decent trade-offs** at the engineering level? Can you clearly articulate, for every technical choice, "why this one"?

In short: this isn't "the barrier dropping," it's "the barrier shifting from coding ability to depth of thinking." I later compressed this judgment into a pair of words: **knowledge is content — AI supplies it infinitely; your judgment, taste, and scars are context — and that's the moat.** The expanded version is in ["From Internship Interviews to Agent Engineering"](https://www.joyehuang.me/en/blog/20260615---frominternshipinterviewstoagentengineering/post).

**Fourth, your opponent isn't other people — it's the you from last year who didn't take action.** What's truly "competitive" in the Agent field isn't your knowledge reserves, it's your "volume of doing." Reading 100 blog posts is worth less than writing one small Agent that actually runs yourself. **Even if you only start today, as long as you take action, you're already outrunning the 90% of people who "watch but don't do."**

### 3.4 Seven common misconceptions

These are the misconceptions I run into most often in consulting; let me debunk them one by one —

**Misconception 1: "I'm bad at math, I can't do AI."** What you want to do is **AI applications**, not **AI algorithms**. Engineering practice at the application layer basically doesn't need math.

**Misconception 2: "I have to finish learning LLM theory before I can learn Agents."** Backwards. The application layer and the underlying algorithms are **two relatively independent tracks**. Do applications first, and go back to fill in theory when you hit a specific problem — it's 10x more efficient.

**Misconception 3: "Doesn't learning this require knowing a lot of frameworks?"** A framework is a tool, not a goal. Once you understand the essence, you can pick up any framework in minutes; if you only know how to use a framework but don't understand the underlying layer, you'll be completely helpless the moment you hit a scenario the framework doesn't support.

**Misconception 4: "I can't find a job without a big-company background."** This Agent direction happens to be **the domain of startups**. What they value is "can you do the work right away" — your project experience, GitHub, and blog matter more than your school and your last employer.

**Misconception 5: "AI is moving so fast, will what I learn be obsolete immediately?"** What changes is the surface vocabulary; what stays is the underlying objects. **The Agent Loop (ReAct is just its 2022 name), context engineering, memory systems, tool calling, Eval systems** — the words keep rotating (the Prompt → Context → Harness → Loop relay is the living proof), but these engineering objects haven't changed in years and won't change in essence in the next five.

**Misconception 6: "Isn't Agent already a red ocean?"** The truly "crowded" fields are the traditional ones that have been deeply developed for twenty years. Agent penetration in fields like Coding, Research, Customer Support, BI, and Marketing is still under 10% — we've barely scratched the surface.

**Misconception 7: "Multi-Agent just means piling on more Agents — the more the stronger."** (New in v2.) The industry paid tuition for this idea in 2025 — Cognition's famous "Don't Build Multi-Agents" was the reflection of that moment. By 2026 the consensus swung back to a clear pattern: **multi-agent works, but the shape is "one orchestrator + isolated subagents + single-threaded writes + subagents return only summaries."** Cognition itself published a revision of its stance in April; Anthropic built a 100,000-line C compiler with 16 parallel Claudes using exactly this pattern. What it means for a newcomer: make a single Agent reliable first; multi-agent is an **orchestration problem**, not a quantity problem.

## Chapter 4 | How to Get Started and Prepare for the Job Hunt

> This is the most hands-on chapter of this document. Everything you need to do — from "opening your IDE and writing your first LLM call" to "landing your first offer" — is here. v2.0 adds prerequisites, minimum deliverables, acceptance criteria, and a budget to every stage — so "where exactly am I" becomes something you can verify yourself.

### 4.1 Prerequisite skills: what you need to know, and what you can skip

**Let's start with languages.** Agent development is currently mainstream in two ecosystems — **Python and TypeScript / JavaScript** — and **knowing one of them already is enough to start**:

- For projects that lean backend, data, or algorithm integration, Python is more common.
- For projects that lean frontend or toward web product forms, TypeScript is more mainstream.
- In my own work, complex Agent projects are usually **a Python backend + a TS frontend** — knowing a bit of both is the most comfortable, but it's not required.

**What you must know:** either Python or TypeScript, with the ability to fluently write functions, handle JSON, and call an HTTP API.

**What you don't need to know (set it aside for now):** deep learning math, PyTorch / TensorFlow / model training / fine-tuning, the internals of the Transformer, and the APIs of Agent frameworks like LangChain / LangGraph (**your first Agent should not start from any of these**).

**On Git and the command line:** these are an engineer's "basic hygiene," but their learning curve has been dramatically flattened by AI tools — when you hit a specific operation you don't know, ask Cursor / Claude Code and it'll walk you through it step by step. **Don't feel "not ready to start learning Agents yet" just because you're unfamiliar with Git — that's putting the cart before the horse.**

**Catch-up list for absolute beginners (starting point A; plan for 1–3 weeks):**

- **Python**: the first six chapters of the official Tutorial, or Harvard's CS50P (free) — goal: functions, dictionaries, exception handling.
- **JSON + HTTP**: MDN's "JSON" and "An overview of HTTP" articles — goal: know what a request, response, status code, and request body are.
- **Git**: GitHub Skills' "Introduction to GitHub" (interactive) — goal: clone / commit / push.
- **Command line**: cd / ls / running a script / setting environment variables (you'll need one for your API key).

How much catch-up is enough? A self-testable bar: **read a JSON file in, change one field, write it back, and push that code to your own GitHub repo** — clear that, and you're ready for Stage 1.

### 4.2 A three-stage learning roadmap

First, restating a v1 judgment that the past year has only strengthened: I do **not** recommend that newcomers hand-write a ReAct Loop from scratch — that rite of passage made sense three years ago, but today's official SDKs are light enough that their docs are the best teaching material.

**Stage 1: Get an SDK running (3–5 days)**

If you're a complete beginner, the only goal for week one is to **manually get a single LLM API call working**. **The biggest asset you'll own at the end of this week isn't code — it's the visceral sense that "an LLM isn't a black box; it's just an HTTP service you can call."**

Then I strongly recommend going straight to the **Vercel AI SDK's official docs** — the docs themselves are an excellent piece of "Agent teaching," progressing layer by layer from "call the model once" to streaming, tool calling, multi-step loops, and structured output. Why not start with the OpenAI Cookbook or Anthropic Cookbook? Because each of those only covers its own model, whereas the **Vercel AI SDK is model-agnostic and has the lowest migration cost**.

The Python route: use the **OpenAI SDK + base_url pattern** to connect to OpenAI-compatible Chinese models, or use the **Anthropic SDK** to connect to Claude. Pydantic AI is the lightweight Python counterpart to the Vercel AI SDK.

> **Prerequisites**: the catch-up list cleared (functions, JSON, calling an HTTP API).
>
> **Tasks**: sign up for the DeepSeek API (V4-Flash at $0.14/$0.28 per million tokens — you won't manage to spend $5) → send "hello" to the model in under 10 lines of code → add multi-turn conversation → follow the Vercel AI SDK docs through streaming and your first tool call.
>
> **Minimum deliverable**: a command-line script that holds a multi-turn conversation, on GitHub, with a README that says how to run it.
>
> **Acceptance criteria**: without looking anything up, you can answer — why does the full messages array get sent back every turn? What is "the model remembers the conversation" actually made of?
>
> **If you're stuck**: paste the raw error into an AI and ask (this is a concrete question — AI is excellent at these); check your API key and base_url; diff your code line by line against the official quickstart.

**Stage 2: Understand the mechanics (1–2 weeks)**

Add a few real tools to the Stage 1 Agent — web search (Tavily / SerpAPI), file read/write, third-party APIs. Focus on observing the model's **"selection behavior" under multiple tools**: when does it pick the wrong one? when does it fall into an infinite loop? Write down every failure case — these notes are the best résumé material.

Then run two comparison experiments: try **LLM Wiki / agentic search mode** once (organize a body of material into Markdown and stuff it into the system prompt, or let the model grep on demand), and then try **RAG mode** once (vectorize and chunk the same material with pgvector / ChromaDB). **Doing this comparison with your own hands is worth more than reading 10 blog posts.**

> **Prerequisites**: the Stage 1 deliverable runs.
>
> **Tasks**: add 2–3 real tools → deliberately collect 10 failure cases (wrong tool picked, infinite loop, bad parameters) → complete the two comparison experiments above.
>
> **Minimum deliverable**: a write-up titled "How My Agent Failed" (publish it — your first public output), with the results and your felt conclusions from the comparison experiments.
>
> **Acceptance criteria**: you can draw the full lifecycle of one tool call (schema definition → model outputs call intent → program executes → result folded back into context); you can state under what conditions your Agent most often picks the wrong tool.
>
> **If you're stuck**: failure isn't an obstacle here — failure cases ARE this stage's output; for a genuinely stuck problem, build a minimal reproduction and take that to ask.

**Stage 3: A real project (1–2 months)**

Pick **a real scenario you yourself would use every day** — don't build a played-out project like a "general-purpose Q&A assistant."

The standard for judging whether a project is "good enough" — refer to W's soul-searching question in my "Mock Interview" post: **"If it were me, I could solve this directly with Doubao / ChatGPT — so why does this have to be built?" If you can't answer, please change the scenario.**

A few entry-level project directions with "résumé value": an AI topic-selection assistant (scraping trending content from Xiaohongshu / Twitter for topic suggestions), a personal Newsletter assistant (auto-summarizing your subscriptions weekly), a simple Chat-to-SQL, a personal email-classification Agent, a lightweight code-review assistant (hooked up to a GitHub Webhook).

> **Prerequisites**: Stage 2 complete, and a scenario that can answer "why not just use off-the-shelf ChatGPT."
>
> **Tasks**: ship a minimum usable version within two weeks → write 5–10 evals for it (input + expected behavior; human scoring is fine, don't chase automation yet) → get at least one real user (a friend counts) actually using it → iterate on feedback.
>
> **Minimum deliverable**: a project someone else can genuinely use, plus a retrospective blog post ("what I built / what broke / how I fixed it").
>
> **Acceptance criteria**: it survives W's soul-searching question; the retrospective contains at least one improvement with a number attached (even a small one like "capped retries at 3 instead of unlimited — infinite-loop cost down 90%").
>
> **If you're stuck**: cut the scope in half. Then cut it in half again — beginner projects die of ambition, not of simplicity.

**Budget for the whole roadmap**: doing all three stages on DeepSeek will almost certainly cost under **¥100 (about $14) in API fees**; add ¥100–200 if you want to feel the Claude / GPT flagships in Stage 3. **Being broke is not a barrier to entering Agents.**

**A parallel path (strongly recommended)**: from Stage 1 onward, **use a mature harness (Claude Code or Codex) daily, in parallel**. Don't just let it write code for you — observe it: how does it organize context? how does it ask you for permissions? when does it compact history? how are its tools described? This is the most industrial-grade harness textbook you can watch for free, and it cross-references beautifully against the small Agent you're hand-rolling.

After finishing each project, **write a blog retrospective** — "what I built," "what pitfalls I hit," "how I solved them." **This blog post is itself the strongest material for your résumé.**

### 4.3 Job-hunt prep: tell your project "to the extreme"

If you can only spend your prep time on one thing, **it's pitching your project to the extreme** — making four things clear:

- **What you did** (What): the project background, your role, the overall architecture
- **Why you did it this way** (Why): the rationale behind every key decision
- **What pitfalls you hit** (How it failed): failure cases + solutions
- **What you learned** (What you learned): how you'd redo it

**Counter-example**: "I used LangChain to build a RAG customer-service system." — a description like this says nothing; it's an interview killer.

**Positive example**:

> "I built a customer-service RAG system. At first I used simple vector retrieval, and recall was only 60% — analysis showed customer questions were colloquial and worded very differently from the source documents. We introduced Query Rewriting: first use a lightweight model to rewrite the user's question into multiple candidate Queries, then retrieve each separately and merge with deduplication. This change lifted recall to 85%, but Token cost went up 30%. To balance the cost, we later added caching — the rewrite results for the same class of questions could be reused. In the end, while keeping recall above 80%, cost only went up 5%."

This passage has: **a metric (60% → 85%) + a decision (Query Rewriting) + a trade-off (cost vs. recall) + a follow-up optimization (caching)**. That's what "telling it to the extreme" means.

**Three points at the résumé level**: don't pile up technical terms ("proficient in LangChain, LangGraph, Vercel AI SDK, CrewAI…" — a résumé like that is laughable; the genuinely strong candidates actually have fewer technical words on their résumés); structure each project entry as "problem — solution — result"; quantify your results (even an estimate beats no number).

**At the interview level** — Agent roles don't test rote memorization, they test "what you've been through." Four concrete dimensions:

1. **Foundational understanding**: the essential differences between LLM / Agent / Chatbot, how Function Calling works, what MCP is…
2. **System design**: context engineering approaches, memory layering, tool-calling reliability, multi-Agent collaboration…
3. **Engineering trade-offs**: the basis for model selection, balancing cost and quality, judgment in framework selection, failure-retry strategies…
4. **Industry awareness**: the design philosophies of Claude Code / Codex / OpenClaw, what "harness" and "loop" are actually about, what you've read lately, which open-source projects you follow…

The first layer relies on experience, the second on understanding, the third on judgment, and the fourth on taste and how much you read. **The further down you go, the more it separates candidates.** In my "Mock Interview" post I gave concrete examples for each category — go take a look if you need them.

### 4.4 Five detours not to take

**Detour 1: gnawing on the LangChain source code right away.** The design is complex and the source is extremely unfriendly to newcomers. Once you've shipped a few projects with an SDK and then go look at it, the experience will feel completely different.

**Detour 2: rushing into model fine-tuning too early.** 99% of application scenarios don't need fine-tuning; prompt engineering + agentic search / RAG already solves most problems.

**Detour 3: chasing new frameworks without building your fundamentals.** There's a new framework every two weeks. Once you form a "chase the new" habit, you'll forever be learning new things and never have a project of your own.

**Detour 4: thinking you've got it just from finishing a tutorial.** In this Agent field, every concept that "looks simple" turns out to have a pile of details once you actually do it. **Watching without writing equals zero.**

**Detour 5: doing without producing output.** Building a project but not writing docs, not writing a blog, not open-sourcing it — that's as good as not having done it. **Output is the most effective way to force input**, and it's also your strongest differentiating asset when you later look for a job.

## Chapter 5 | The Few Things That Truly Matter

> The first four chapters made clear "what it is, how to start, how to get a job." This last chapter is for those who've already finished their first project and want to know "what does going deeper look like" — and it's the true inner skill of an Agent engineer.
>
> Each section uses an everyday analogy to help you build intuition. After reading this chapter, you'll have a shared language for talking with senior engineers.

### 5.1 Context engineering

**Analogy**: when you hand off work to a colleague, do you give them a 100-page project archive, or a 1-page concise brief?

An LLM's attention is finite — the longer the context and the lower the information density, the more easily it "loses focus," and at the same time Token cost goes up and responses slow down. **Context Engineering is exactly about "presenting the information that most deserves to be seen, in the most effective way, within a finite space."**

In a real project, a commercial-grade Agent might process dozens of interactions and call dozens of tools in a single conversation. Without context management, you'll blow the context out within 10 minutes. Common techniques —

- **Structured Prompt**: use XML tags, JSON blocks, and clear delimiters instead of a stream-of-consciousness natural-language dump.
- **Front-loading / back-loading key information**: the model pays more attention to the beginning and the end (the "Lost in the Middle" phenomenon). Put important constraints at the top of the System Prompt or at the end of the User message.
- **Replacing verbatim history with a summary**: compress early conversation in a long dialogue into a summary (since 2026 mainstream APIs offer server-side automatic compaction, e.g. Anthropic's compaction).
- **Prefix-Cache-friendly context design**: put unchanging content first and changing content last, which can massively reduce cost.

### 5.2 Memory systems

**Analogy**: how do people remember things? Short-term memory (things that just happened), long-term memory (important experiences from years ago), retrieval cues (seeing an old photo and suddenly recalling a story). An Agent's memory architecture basically mimics this.

**Three-layer architecture**:

- **Working Memory**: the context the current task is actively using
- **Short-term Memory**: the history of the current session
- **Long-term Memory**: persistent information across sessions

There are three key decision points for long-term memory —

**Write strategy**: what kind of information is worth writing? A temporary preference like "I feel like eating spicy today" shouldn't be remembered; a long-term fact like "I'm allergic to peanuts" must be. This classification is usually judged by a dedicated "Memory Agent."

**Read strategy**: when to retrieve, and how? Retrieve once on every turn or only under a specific intent? Use vector similarity, keywords, or graph retrieval?

**Forget strategy**: more long-term memory isn't better. Stale, low-value, or contradictory memories should be cleaned up or decayed.

And one practical observation from 2026 (also mentioned in 2.4): **the long-term memory designs that win in practice are mostly "file-based + agent-managed"** — Anthropic's memory tool, CLAUDE.md, the LLM Wiki are all this shape. When you build a memory system, don't reach for a vector database as step one by default.

### 5.3 Tool calling

**Analogy**: getting a smart but handless person to complete a task for you — you have to tell them which tools are nearby, what each one does, and how to use it.

A few common engineering difficulties:

- **Tool Schema design**: the clearer you write the parameter names and descriptions, the lower the chance the model misuses them.
- **Trade-off on the number of tools**: too few isn't enough, too many and the model can't pick correctly. **"Around 10" is a common rule of thumb, not a law** — the real ceiling depends on the model and how distinguishable your tool descriptions are; beyond it you need "tool routing" (exactly the problem Skills' on-demand exposure solves).
- **Failure retry and idempotency**: retrying on failure is necessary, but it needs a cap — **"3 tries" is a common heuristic**; the right retry policy depends on the error type and idempotency: a timeout is retryable, "insufficient balance" won't fix itself in ten thousand retries.
- **Prompt constraints and execution-layer permissions are two different gates — install both** (v2 rewrote this item; v1's phrasing was a safety hazard): making tool-usage boundaries clear in the prompt significantly reduces mis-calls and saves a lot of cleanup cost — but it is **not a security boundary**. Any tool with side effects (placing orders, sending messages, writing to a database, deleting files) must have hard controls at the **tool-execution layer**: allowlists, parameter validation, amount/scope caps, pre-execution confirmation, idempotency keys, audit logs. In one sentence: **the prompt makes errors rarer; the program makes them impossible — the model's self-restraint is never a security mechanism.**

### 5.4 Reliability

**Analogy**: writing a Demo is like cooking in your own kitchen; writing Production is like running a restaurant — what you have to handle isn't just "how good the food is," but also "will it blow up at peak hours" and "will the occasional picky customer break the process."

A traditional application is (at least at the business-logic layer) approximately a "deterministic system" — the same input yields the same output. An Agent is a "probabilistic system" — the same input may yield different outputs, or even fail outright. **This means the "test it once and it's OK" development model completely fails to work for Agents.**

Common reliability problems: hallucination, instruction drift, unstable formatting, infinite loops, and cascading collapses from tool failures.

The core engineering ideas:

- **Up-front constraints**: use the prompt to make "how it should be done" clear — lower cost than an after-the-fact fallback (note: this is about reducing errors, and does not replace the execution-layer permission control of 5.3)
- **Structured output + Schema validation**: validate model output with Pydantic, Zod
- **State machine + Checkpoint**: make the Agent flow explicit as a state machine
- **Degradation strategy**: have a fallback path when a tool fails

### 5.5 Cost control

**Analogy**: driving — gas prices, distance, and the car model all affect the fuel bill. An Agent is the same: the model, context length, and number of calls together determine the cost of a single task.

An Agent's cost is far higher than a traditional application's. A single complex task might take dozens of LLM calls and accumulate tens of thousands to hundreds of thousands of Tokens — a single task could cost anywhere from a few yuan to a few dozen yuan. **If your product is free and toC, poor cost control means losing money for the applause.**

A few high-ROI optimization techniques:

- **Prefix-Cache-friendly design**: mainstream vendors give steep caching discounts for "prefix hits" (Anthropic's cache reads cost roughly one-tenth of base price; check each vendor's pricing page for specifics). Put unchanging content first.
- **Tiered model usage**: use a cheap model for simple tasks, and only use the flagship for complex ones.
- **Reducing the number of Agent Steps**: don't split something that can be said clearly in one step into multiple steps.
- **Context pruning**: remove irrelevant tool results and stale conversation history from the context.

### 5.6 Evaluation (Eval)

**Analogy**: traditional software can use unit tests — input 1+1, expect output 2, and if it's wrong, it's a Bug. On many tasks an Agent has no single "standard answer" — so how do you know it did "well"?

But first, a correction of something v1 stated too strongly: an Agent "having no standard answer" **does not mean "having no decidable constraints"** — booking the wrong ticket, leaking data, calling a forbidden tool are all unambiguously wrong. So an Eval system has to cover two layers at once: **hard constraints (tests that must all pass) + a quality spectrum (the scored, comparative part)**. With that machinery in place, you can finally answer "is my new version of the Agent better or worse than the last one?" — and without it, you can optimize all day with no idea whether you're even pointed the right way. **The Eval system is the marker of Agent engineering going from "workshop" to "industry."**

Mainstream evaluation methods:

- **Offline evaluation**: prepare a batch of test cases, run the Agent, and score by hand or with LLM-as-Judge
- **Online evaluation**: collect real user feedback in production (thumbs up / down, dwell time, whether they keep following up)
- **LLM as a Judge**: use a stronger model as the judge — but watch out for its own biases (a tendency to score high, a preference for long answers, etc.)
- **Controlled experiments**: A/B Test, splitting the new and old versions to different users

**A 2026 industry footnote: Eval went from "inner skill" to "hot commodity."** OpenAI's CPO says writing evals is the most important skill for a product manager in the AI era; OpenAI outright acquired the eval company Promptfoo in March; the eval platform Braintrust raised an $80M Series B; and Tencent Research Institute's "Age of the Super Individual" report lists "evaluation" among its ten keywords for AI-native work. **For a job seeker, "can build evals" is currently the best value-for-effort differentiator — building an eval suite for your internship project or your own project is the most concrete overtaking-lane advice I can give** (expanded in the "go do evals" section of ["From Internship Interviews to Agent Engineering"](https://www.joyehuang.me/en/blog/20260615---frominternshipinterviewstoagentengineering/post)).

### 5.7 These six things are an Agent engineer's true "inner skill"

To sum up —

1. **Context engineering**: maximize information density within a finite space
2. **Memory systems**: let the Agent remember things in layers, like a person
3. **Tool calling**: let the Agent "act" — and not run wild
4. **Reliability**: switch from deterministic thinking to probabilistic thinking
5. **Cost control**: the money really does burn
6. **Evaluation**: Agent optimization without Eval is all guesswork

If you can clearly articulate these six things in your résumé or interview, you're already ahead of most applicants.
